이 시리즈 내내 LLM 심판 얘기를 했지만, 심판을 믿을 근거는 결국 사람이다. 같은 동화 쌍을 사람도 평가해서 심판과의 일치도(κ)를 재는 골든셋. 그 사람 평가에서 두 가지 문제를 겪었다. 하나는 설계 문제(어떤 쌍을 보여줄 것인가), 하나는 사고(평가한 게 사라졌다)다.

## 797쌍 중 30쌍, 어떻게 골라야 공정한가

사람이 수백 쌍을 다 볼 수는 없다. 앱에는 원래 대표 표본 n쌍만 평가하는 기능이 있었고, (모델쌍 × 단어수 구간)으로 층을 나눠 라운드로빈으로 뽑았다(2편). 실측 데이터로 다시 보니 구멍이 있었다. 주제가 층에 없었다.

이 시스템의 프롬프트는 주제×테마×길이 로테이션으로 생성된다. 프롬프트 하나는 주제와 단어수가 고정이라는 뜻이다. 층 목록을 삽입 순서대로 돌면 작은 n에서는 앞쪽 프롬프트의 주제와 길이에 표본이 쏠린다. 모델은 고루 섞였는데 전부 용 이야기가 될 수 있는 구조였다.

수정은 두 가지였다.

1. 주제를 층 차원에 추가. 층 키를 (모델쌍 × 주제 × 단어수 구간) 3차원으로.
2. 층 순서를 결정론적으로 셔플. 고정 시드 Fisher-Yates로 층 배열을 섞은 뒤 라운드로빈. 시드가 고정이라 결정론은 유지된다.

알고리즘 전체는 몇 줄이면 적힌다.

```text
strata ← 전체 쌍을 key = (sort(모델a,모델b), 주제, 단어수구간) 로 그룹화
strata를 key 사전순 정렬          # 삽입 순서 의존 제거
strata를 고정 시드로 셔플          # 주제·길이 군집 해체 (결정론 유지)
i ← 0; 선택 ← []
while |선택| < n:
    for 각 층 in strata:          # 라운드로빈
        층에 i번째 쌍이 있으면 선택에 추가; n 차면 종료
    i ← i + 1
```

라운드로빈이라 각 층에서 비례만큼 나오고, 셔플이 앞쪽 n개가 어느 층부터 오는지를 흩는다.

셔플이 왜 필요한지가 이 수정의 요점이다. 층화 라운드로빈은 각 층에서 고르게 뽑는 것은 보장하지만, 앞쪽 n개가 어느 층부터 오는지는 층 순서가 정한다. 주제와 길이가 프롬프트에 묶여 있는 구조에선 순서 자체를 흩어야 작은 n이 3차원을 고루 덮는다.

결정론을 지킨 이유는 평가자 간 비교다. 난수 없이 같은 데이터에 같은 n이면 항상 같은 표본이 나와야, 여러 평가자가 같은 쌍을 보고 그 위에서 κ와 α를 잴 수 있다. 표본을 30에서 50으로 늘리면 앞의 30쌍이 그대로 유지되는 중첩성도 같은 이유로 필요하다.

797쌍 실측 데이터로 검증한 결과는 이렇다.

| n | 모델 커버 | 주제 커버 | 단어수 구간 커버 |
|---|---|---|---|
| 30 | 13/13 | 11/13 | 5/5 |
| 50 | 13/13 | 11/13 | 5/5 |
| 100 | 13/13 | 11/13 | 5/5 |

30쌍만 뽑아도 13개 모델 전부와 5개 길이 구간 전부가 들어온다. 결정론(같은 n이면 같은 표본)과 중첩(n=30 ⊂ n=50)도 확인했다.[^strata]

이름을 정확히 해두면 이건 stratified k-fold 교차검증이 아니다. k개 폴드를 회전시키는 CV가 아니라, 각 폴드가 비율을 보존한다는 층화 원리를 대표 부분집합 1개 추출에 적용한 것이다. 층화 표집(stratified sampling)이고, 문서에도 "계열"이라고만 적었다. 용도가 사람에게 보여줄 대표 표본 뽑기라서 이게 맞는 도구다.

## 사고: 평가한 투표가 사라졌다

표본 30쌍을 골라 평가를 진행하던 중이었다. 10쌍쯤 투표한 상태에서 다시 열어보니 평가가 사라져 있었다.

내 데이터가 날아간 사고라 원인 추적을 제대로 했다. 앱 코드에서 투표(`STATE.human`)를 지우는 경로를 전부 뒤졌다. 있었다. 실측이 완료되거나 데모를 재생성할 때, 그리고 저장된 쌍과 현재 모델 목록이 안 맞을 때 상태를 리셋하면서 `STATE.human`을 비우는 경로들이다. 새 결과를 받을 준비를 하는 코드가 이전 사람 투표까지 같이 청소하고 있었다.

불행 중 다행이 하나 있었다. 투표는 저장될 때 `STATE.human`(현재 평가자 화면용)과 `STATE.raters[평가자].votes`(평가자별 누적) 두 곳에 미러되는데, 문제의 리셋 경로 대부분이 human만 지우고 raters는 남겨뒀다. 데이터가 완전히 죽은 게 아니라 화면에 안 보이는 곳에 살아 있을 가능성이 있었다.

복구를 이렇게 만들었다.

- 앱을 켤 때 `STATE.human`이 비었는데 raters나 IndexedDB 백업에 현재 쌍과 맞는 투표가 남아 있으면 자동 복원하고, 안 한 쌍부터 이어하기 지점을 잡는다.
- 이 시나리오를 그대로 단위 테스트로 고정했다. 30쌍 중 10표가 raters에만 남은 상태에서 복구 후 human에 10표, 다음 평가 위치는 11번째. 9개 케이스 전부 통과.[^recover]

이 글을 쓰는 시점에 그 10표가 내 브라우저에서 실제로 되살아났는지는 아직 확인하지 못했다. 에이전트를 시켜 브라우저 밖에서 Chrome의 로컬 저장소 파일(leveldb)을 직접 뒤지는 것까지 해봤지만 값이 압축 블록 안에 있어 안전하게 꺼내지 못했다. 복구 장치는 갖춰졌고, 데이터가 남아 있었다면 자동으로 돌아온다. 남아 있지 않았다면 아래 시스템이 없던 시절의 수업료다.

## 3중 저장과 GitHub 백업

사고의 교훈으로 사람 투표에 저장 계층을 깔았다.

```
투표 1건 발생
 ├─ localStorage        (즉시 · 빠름 · ~10MB 한도)
 ├─ IndexedDB           (즉시 · 사실상 무제한 · 브라우저 재시작에도 유지)
 └─ GitHub              (버튼 1클릭 · eval/runs/human/votes-<평가자>.json 커밋)
앱 시작 시: human 비었으면 raters → IndexedDB 순으로 자동 복구
리셋 경로: raters(사람 투표)는 더 이상 지우지 않음
```

GitHub 백업에는 목록 UI도 붙였다. 평가 화면에 들어가면 저장소의 백업들(평가자, 표수, 날짜)이 자동으로 나열되고, 골라서 이어하거나 새로 시작할 수 있다. 다른 컴퓨터에서도 자기 백업을 눌러 안 한 쌍부터 계속하면 된다. 실측 결과(4편)처럼 사람 투표도 GitHub 저장소가 백엔드가 됐다.

돌아보면 4편의 localStorage 침묵 실패와 이번 투표 유실은 같은 병의 두 증상이다. 브라우저 단일 저장소를 믿은 것. LLM 판정은 돈이라 일찍 백업을 깔았으면서 사람 투표는 몇 개 안 되니까 하고 미뤘는데, 사람 투표야말로 이 시스템에서 재생산 비용이 가장 큰 데이터였다. API는 다시 돌리면 되지만 사람에게 그 30쌍을 다시 읽어달라고는 못 한다.

## 시리즈를 닫으며

6월 16일 루브릭 문서에서 시작해 한 달 반이 지났다. 숫자로 요약하면 의존성 0 파이썬 파이프라인과 3,400줄 단일 HTML 앱, 스케줄러 시뮬레이션과 $46.76 실측 797쌍, 교차 리뷰로 잡은 결함 5개, 리플레이 검증으로 얻은 절감 25~58%다.

기술적 결론은 각 편에 있으니 과정에서 배운 것만 남긴다. 평가 시스템에서 무서운 건 크래시가 아니라 조용히 틀리는 것이었다. 좁은 CI, 침묵하는 tie, 0이 되는 지표. 보이게 만드는 일(텔레메트리, 검증 스크립트, 근거 전문 공개)이 절반이었다. 정직한 신뢰구간에는 비용이 붙지만 그 비용은 표집을 바꿔서 줄일 수 있었다. 그리고 심판에게 자기평가를 금지시킨 논리는 내 코드에도 적용됐어야 했다.

다음 실측은 자기평가 제외를 켜고, temp 0 심판과 쌍 클러스터 CI로 돌린다. 그 결과가 이번 순위와 얼마나 달라지는지가 지금 제일 궁금하다.
