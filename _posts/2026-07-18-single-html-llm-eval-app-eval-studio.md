[2편](/posts/llm-eval-pipeline-from-scratch-bradley-terry/)에서 파이썬 평가 파이프라인을 만들었다. 그런데 파이프라인을 커밋한 6월 18일, 같은 날 480줄짜리 <span class="term" data-tip="웹 문서의 구조와 의미를 요소로 표시하는 마크업 언어. CSS는 표현을, JavaScript는 동작을 주로 담당한다.">HTML</span> 파일을 하나 더 커밋했다.[^v1] 한 달 뒤 이 파일은 3,400줄이 됐고, 프론티어 모델 13개의 실측 평가 전체를 브라우저 안에서 돌리는 엔진이 됐다.

같은 걸 왜 두 번 만들었는지, 그 결정이 어떤 문제를 불렀는지가 이번 편이다.

## 파이썬으로 안 되는 게 하나 있었다

이 평가 시스템에서 <span class="term" data-tip="Large Language Model. 많은 텍스트에서 토큰의 조건부 분포를 학습해 문장을 생성하거나 분류·요약·추론 작업을 수행하는 언어 모델을 뜻한다.">LLM</span> 심판의 순위는 절반이다. 나머지 절반은 사람 <span class="term" data-tip="사람이 직접 평가한 소량의 기준 데이터. 같은 항목을 자동(LLM) 평가와 사람이 모두 평가하게 한 뒤 일치도를 재면, 자동 평가를 얼마나 믿어도 되는지가 숫자로 나온다.">골든셋</span>이다. 같은 동화 쌍을 사람도 평가해서 심판과의 일치도(Cohen's κ)를 재야 "심판을 믿어도 된다"가 성립한다.

문제는 그 사람이 나 혼자가 아니라는 것. 멘토님, 팀원, 나중엔 학부모까지 평가에 참여할 수 있어야 했다. 그분들께 Python을 설치하고 터미널에서 스크립트를 돌려달라고 할 수는 없다.

그래서 요구사항을 이렇게 정했고, 이게 곧 아키텍처가 됐다.

> 파일 1개. 빌드·서버·의존성 없음. 더블클릭하면 바로 열린다.[^appreadme]

React도 빌드 도구도 안 썼다. 단일 `index.html`에 순수 JS로 UI와 통계 엔진(Bradley-Terry, 부트스트랩, κ)을 전부 넣었다. 메신저로 파일 하나 보내면 상대방 브라우저에서 그대로 돈다. 배포가 곧 파일 전송인 앱이다. <span class="term" data-tip="소프트웨어가 다른 소프트웨어의 기능이나 데이터에 접근할 때 따르는 호출 계약. 사용할 주소, 입력 형식, 인증, 응답과 오류 규칙을 함께 정의한다.">API</span> 키가 없어도 고정 시드 합성 데이터로 전체 흐름이 도는 데모 모드를 기본으로 깔아서, 처음 여는 사람이 설정 없이 30초 안에 평가 방식을 체험할 수 있게 했다.

트레이드오프는 알고 시작했다. 단일 파일은 자랄수록 모놀리스가 된다. 지금 3,420줄인데, 이 결정의 비용은 6편에서 정리한다.

완성형 기준의 전체 구조를 먼저 놓고 시작하는 게 이해에 빠를 것 같다. 아래가 이 앱의 실측 경로 전부다. 이후 편들은 이 그림의 각 상자가 왜 그렇게 생겼는지에 대한 이야기이기도 하다.

그림을 왼쪽 위부터 읽으면 이렇다.

> **2026-07-22 후속 변경.** 아래 개발기는 당시 `aligned-v2` 기준에서 출발했다. `feat/eval-prompt-provenance`의 `childlit-v3`에서는 창작 본선이 주제·형식·길이·독서 방식·연령만 받고, 수치 Lexile은 기본 OFF인 독립 난이도 프로브로 분리된다. 새 설계와 아직 남은 검증은 [프롬프트 재설계 근거](/posts/creative-ranking-reading-level-two-track/)에 기록했다.
{: .prompt-warning }

- **설정** — 평가 대상 모델 N개와 심판 <span class="term" data-tip="서로 다른 회사 계열의 심판 여러 명을 함께 쓰는 구성. 심판마다 취향과 편향이 다른데, 계열을 섞으면 한 심판의 성향이 순위를 좌우하는 걸 줄일 수 있다.">jury</span>, 동화 권수, 예산 상한, 스케줄(full 또는 <span class="term" data-tip="이 프로젝트의 은퇴 스케줄러. 배치(5권)마다 BT와 CI를 갱신하고, CI 분리가 2연속 확인된 모델만 은퇴시킨다. 한 번의 우연으로 탈락시키지 않는 확인 절차가 이름의 유래다.">retire2</span>)을 고른다. 실측 시작 전에 선택된 모든 모델에 테스트 호출을 날려 모델명(slug)이 실제로 존재하는지 먼저 확인한다. 비싼 호출 전에 싼 점검을 두는 이유는 바로 아래에서 다룬다.
- **생성** — 프롬프트마다 생존 모델 각각이 동화를 한 편 쓴다. `childlit-v3` 본선 user prompt에는 주제·형식·길이·독서 방식·목표 연령을 넣고 수치 <span class="term" data-tip="영어 독자의 읽기 능력과 텍스트 난이도를 같은 척도에 표시하는 상용 지표. 텍스트 측정은 어휘의 빈도와 문장 길이 같은 특성을 사용하며, 이 프로젝트의 가독성 공식 합성값은 공인 Lexile이 아니다.">Lexile</span>은 넣지 않는다. 요청 길이에 맞춘 출력 <span class="term" data-tip="모델의 토크나이저가 텍스트를 나눈 처리 단위. 한 토큰은 단어 하나와 같지 않으며 같은 문장도 모델별 토크나이저에 따라 토큰 수가 달라질 수 있다.">토큰</span> 상한을 두고 창작에는 <span class="term" data-tip="다음 토큰 확률분포의 평탄함을 조절하는 값. 낮을수록 높은 확률의 토큰에 집중하지만 temperature 0도 제공자·모델·실행 환경에 따라 완전한 재현성을 보장하지 않는다.">temperature</span> 0.7을 쓴다. 이미지·그림 지시·페이지 메모는 생성 실패로 제외하고, 사람 검토용 한국어 번역은 순위에 쓰지 않는다. 수치 Lexile은 별도 <span class="term" data-tip="아이가 어른의 실시간 도움이나 낭독 없이 직접 글자를 해독하고 내용을 이어 가는 읽기 방식. 같은 아이도 읽어주는 글을 이해하는 수준과 혼자 읽는 수준이 다를 수 있다.">독립 읽기</span> 프로브에서만 사용한다.
- **판정** — 생존자끼리의 모든 쌍을 심판마다 두 번(A,B와 B,A) 묻는다. 판정의 샘플링 변동을 줄이기 위해 temperature 0을 쓰지만 완전한 결정성을 가정하지 않는다. 심판이 후보와 같은 계열이면 그 쌍에서 빠진다. 응답에서 승자를 못 읽으면 몰래 무승부로 넘기지 않고 실패로 집계해 화면에 보여준다.
- **배치 경계(retire2)** — 5권마다 <span class="term" data-tip="맞대결 승패만으로 각 후보의 숨은 실력을 추정하는 통계 모델(1952). 실력 차가 승률을 정한다고 가정하고, 관측된 모든 승패를 가장 잘 설명하는 실력값을 최우도로 찾는다. 경기 순서와 무관하게 같은 답이 나오는 배치 방식이라, 고정된 대전 기록의 순위에 적합하다.">Bradley-Terry</span>를 다시 적합하고 쌍 클러스터 <span class="term" data-tip="가진 데이터에서 복원추출로 여러 번 가짜 표본을 만들어 같은 계산을 반복하고, 그 결과들의 흩어짐으로 추정치의 불확실성을 재는 방법. 표본이 모집단을 대표한다면 재표집의 흔들림이 실제 추정 오차와 비슷하다는 원리다.">부트스트랩</span>으로 95% <span class="term" data-tip="Confidence Interval의 약칭. 이 글에서는 추정 불확실성을 나타내는 신뢰구간을 뜻하며 CI/CD의 CI와는 다른 용어다.">CI</span>를 갱신한다. CI가 2연속 분리된 모델은 은퇴하고, 은퇴자도 그 시점 점수로 리더보드에 남는다. 생존자만 다음 배치로 간다.
- **결과** — 주 순위는 Arena(<span class="term" data-tip="Bradley–Terry 모델의 약칭. 두 후보의 상대적 실력으로 맞대결 승률을 설명하고 전체 pairwise 결과에서 실력값을 추정한다.">BT</span>)와 CI, 보조로 <span class="term" data-tip="경기가 끝날 때마다 결과와 기대승률의 차이만큼 점수를 즉시 조정하는 체스식 레이팅. 실력이 변하는 선수를 추적하는 데 좋지만, 경기 순서에 따라 최종 값이 달라져 고정된 대전 기록의 순위에는 부적합하다.">Elo</span>. 축별 진단과 자기 계열 편향 카드, 자동 <span class="term" data-tip="글을 얼마나 수월하게 읽고 이해할 수 있는지를 뜻한다. 문장·단어 표면값으로 계산한 가독성 공식은 정량 단서일 뿐 의미 구조, 배경지식, 독자와 읽기 목적 전체를 대신하지 않는다.">가독성</span> 프록시와 선택형 난이도 <span class="term" data-tip="목표 레벨을 한 단계 바꿨을 때 측정 난이도가 실제로 몇 단계 움직였는지의 회귀 기울기. 1.0이면 지시한 만큼 정확히 반응한다는 뜻. 절대 눈금이 틀린 자로도 변화량은 잴 수 있다는 발상이다.">반응성 β</span>까지 한 화면이다. 자동 난이도 수치는 BT·은퇴에 쓰지 않는다. 완료되면 report.html과 results.json이 GitHub에 아카이브된다. 세 경로의 관계는 [2트랙 아키텍처 글](/posts/creative-ranking-reading-level-two-track/)에 따로 정리했다.
- **저장 계층** — 진행 상태는 1.5초마다 <span class="term" data-tip="브라우저가 출처별로 제공하는 동기식 문자열 키-값 저장소. Web Storage 한도는 브라우저에 따라 다르고 한도를 넘으면 QuotaExceededError가 날 수 있어 큰 체크포인트 저장에는 맞지 않는다.">localStorage</span>에 저장되고, 10MB 한도를 넘으면 <span class="term" data-tip="브라우저가 출처별로 제공하는 비동기 객체 데이터베이스. localStorage보다 큰 구조화 데이터를 다루지만 저장 한도와 삭제 정책은 브라우저·디스크 여유·사용자 설정에 따라 달라진다.">IndexedDB</span>로 즉시 폴백한다. 10쌍마다 IndexedDB와 GitHub에 자동 백업도 돈다. 유실 사고를 두 번 겪고 만든 구조다(5편, 8편).
- **사람 골든셋** — 층화 표본으로 뽑은 쌍을 사람이 평가하고, pair_id로 심판 판정과 매칭해 일치도(κ, α)를 잰다.
- **파이썬 미러** — 같은 수식을 파이썬으로도 유지한다. 등가성 테스트가 두 구현이 같은 입력에 같은 순위를 내는지 지킨다.

그림의 각 주장은 [index.html](https://github.com/C0mput33/little-bard/blob/main/eval/app/index.html)에서 확인할 수 있다. 판정의 `{temp: 0}`, 부트스트랩 `iters=1000`, 쌍 단위 재표집, 1.5초 저장, 10쌍마다 백업, 자기 계열 제외 기본값이 한 파일에 있다.

한 파일 안에서 신호 수집, 통계, 영속성, 검증을 분리했다. 신호 수집은 생성·판정 호출과 재시도, 통계는 집계·은퇴, 영속성은 3중 저장, 검증은 사람 골든셋과 파이썬 미러를 맡는다. 이후 수정도 이 경계를 기준으로 설명한다.

## 첫 실측: 시작하자마자 실패

6월 19일, 처음으로 실제 API(OpenRouter)를 붙여 실측을 돌렸다. 실패했다. 원인을 파보니 허무했는데, 모델 목록에 넣어둔 <span class="term" data-tip="API에서 모델을 지정하는 고유 식별 문자열(예: openai/gpt-5.2). 오타가 나면 호출 시점에야 404로 드러난다.">slug</span> 6개가 <span class="term" data-tip="여러 회사의 LLM을 하나의 API와 결제로 호출하게 해주는 중계 서비스. 모델마다 계정을 따로 만들 필요가 없어 다모델 비교 실험에 편하다.">OpenRouter</span>에 존재하지 않는 이름이었다.[^slugfix]

모델 카탈로그를 문서와 기억으로 작성한 게 화근이었다. 존재하지 않는 모델명은 호출 시점에야 404로 드러나고, 그때는 이미 다른 모델 호출에 돈을 쓴 뒤다.

수정하면서 두 가지를 앱에 넣었다.

1. 사전 점검. 실측 시작 전에 선택된 모든 모델에 1토큰짜리 테스트 호출을 날려 slug 유효성을 확인한다. 실패 목록을 한 번에 보여주고 시작 자체를 막는다.
2. 에러 상세화. <span class="term" data-tip="클라이언트와 서버가 요청과 응답을 주고받는 웹 전송 규약. 메서드, 상태 코드, 헤더와 본문 형식을 정의하며 TLS를 사용한 암호화 연결은 HTTPS라고 부른다.">HTTP</span> 상태별로 원인을 사람 말로 번역한다. 401은 키 오류, 402는 크레딧 부족, 404는 slug 없음, 429는 요청 한도.

당연한 걸 왜 처음부터 안 했나 싶지만, "일단 돌려보자"로 시작한 코드가 원래 이렇다. 같은 날 <span class="term" data-tip="같은 두 답을 A/B와 B/A 순서로 각각 평가해 순서에 따른 변화를 확인하는 방법. 두 결과를 함께 쓰면 위치 편향을 줄일 수 있지만 완전히 없어진다고 보장되지는 않는다.">양방향 swap</span>(위치 편향 통제)도 앱에 구현했다.

## 두 언어, 같은 엔진, 첫 드리프트

6월 23일에 찾은 버그가 하나 있다. 같은 판정 데이터인데 앱과 파이썬 파이프라인의 순위가 미묘하게 달랐다. 원인은 swap 집계 방식이었다. 파이썬은 (쌍, 심판)별로 정방향·역방향 판정을 평균해 <span class="term" data-tip="집계의 최소 단위. 한 쌍에 대한 한 심판의 판정 하나(양방향 평균을 마친 점수)를 말한다. 심판 셋이 본 쌍이면 contest 3개가 BT에 들어간다.">contest</span> 하나로 만드는데, 앱은 두 판정을 각각 넣고 있었다.[^swapfix]

평균이어야 하는 이유가 있다. 정방향과 역방향을 별개의 contest 두 개로 넣으면, <span class="term" data-tip="내용과 무관하게 먼저 보여준 답을 더 좋게 평가하는 경향. LLM 심판에게 일관되게 관측되는 대표적 편향이라, 순서를 바꿔 두 번 묻는 통제가 필요하다.">위치 편향</span>이 만든 "정방향 A승, 역방향 B승"이 실력 신호처럼 우도에 두 번 들어간다. (쌍, 심판) 단위로 먼저 평균해 contest 하나(s ∈ {0, ¼, ½, ¾, 1})로 접어야 위치 효과가 집계 단위 안에서 상쇄된다. 파이썬은 처음부터 이 방식이었고 앱이 어긋나 있었다.

앱을 파이썬과 같은 방식으로 통일하고, 두 구현이 같은 입력에 같은 순위를 내는지 확인하는 등가성 테스트를 만들었다. 같은 judgments를 양쪽 엔진에 넣어 <span class="term" data-tip="이 시스템의 최종 점수. BT로 추정한 실력값을 체스 레이팅처럼 읽히도록 평균 1000 기준으로 변환한 것. 두 모델의 점수 차가 곧 승률로 환산된다(차이 100이면 약 64%, 400이면 약 91%).">Arena 점수</span>를 대조하는 방식이다. 같은 엔진을 두 언어로 유지하는 게 공짜가 아니라는 걸 이때 처음 체감했다. 한쪽을 고치면 다른 쪽도 고쳐야 하고, 잊으면 표시 없이 갈라진다. 같은 문제가 6편(심판 temperature)과 7편(부트스트랩 방식)에서도 반복된다.

## 사람 평가 기능의 진화

앱의 존재 이유가 사람 평가였으니 이 부분이 가장 많이 바뀌었다. 전부 6월 23일과 24일 이틀에 몰려 있다.

| 기능 | 왜 필요했나 |
|---|---|
| 다중 평가자 (rater별 누적·전환) | 한 노트북으로 여러 명이 돌아가며 평가하는 상황 |
| 팀 분산 평가 (HTML 배포 → 결과 파일 병합) | 각자 자기 컴퓨터에서 평가하고 jsonl로 합치기 |
| 평가자 신뢰도 종합 (<span class="term" data-tip="평가자 수가 일정하지 않거나 일부 판정이 비어 있어도 계산할 수 있는 신뢰도 계수. 자료 척도에 맞는 불일치 함수를 쓰며 1은 완전 일치, 0은 우연 기대 수준이다.">Krippendorff α</span>) | 2명은 <span class="term" data-tip="두 평가자의 범주형 판정 일치도에서 주변분포로 기대되는 우연 일치를 보정한 계수. 1은 완전 일치, 0은 기대 일치 수준이며 음수도 가능하다.">Cohen's κ</span>로 되지만 N명 일치도는 α가 필요[^alpha] |
| 부분 평가 (대표 표본만) | 쌍이 수백 개면 다 못 본다. 모델×단어수 비율을 유지한 층화 표본 |
| 실측 비용·ETA·예산 상한 | 실측은 돈이라, 예산을 넘으면 자동 중단 |

층화 표본은 난수 없이 <span class="term" data-tip="같은 입력과 상태에서 같은 결과를 내는 성질. 시드 고정은 난수 경로를 통제하는 한 조건일 뿐이며 구현·하드웨어·외부 서비스가 바뀌면 재현이 깨질 수 있다.">결정론적</span>으로 뽑는다. 같은 데이터에 같은 n이면 항상 같은 표본이다. 평가자들이 같은 쌍을 봐야 평가자 간 일치도가 성립하기 때문이다.

상태 설계도 이때 골격이 잡혔다. 앱의 전 상태는 `STATE` 객체 하나로 모이고, 여기서 갈라지는 저장 단위가 곧 데이터 모델이다.

| 상태 | 담는 것 | 분리한 이유 |
|---|---|---|
| `pairs[]` | 쌍별 동화 원문(EN/KO)과 심판별 점수·근거 전문 | 분석과 리포트의 원시 데이터. 근거 전문 보존이 원칙 |
| `human` / `raters` | 현재 평가자 화면용 투표 / 평가자별 누적 투표 | 여러 명이 한 앱을 쓰니 화면과 장부를 분리 |
| `live` | 실측 진행 <span class="term" data-tip="진행 상태를 통째로 저장해둔 지점. 중단되거나 크레딧이 떨어져도 완료분을 다시 호출하지 않고 그 지점부터 이어서 실행할 수 있다.">체크포인트</span>(생성물·판정 캐시·은퇴 상태) | 중단 지점부터 재개하려면 완료분과 진행분이 달라야 함 |
| `usage` / `modelCost` | 콜·토큰·비용 누적, 모델별 편당 비용 | 예산 가드와 품질×비용 분석의 입력 |
| `reviewIds` | 층화 표본으로 고른 쌍 id | 평가 범위 자체도 상태다. 새로고침에도 유지돼야 함 |

이 분리는 뒤에서 두 번 도움이 됐다. 실측이 중단돼도 `live`의 판정 캐시 덕에 완료분을 다시 호출하지 않았고(6월 28일 크레딧 소진 때 실전 확인), 투표가 화면에서 사라졌을 때 `raters` 장부에 살아 있었다(8편).

## 첫 완주와 GitHub 백엔드

6월 24일에 처음으로 실측을 완주했다. 8모델 × 5심판, 84/84쌍.[^firstrun] 완주하고 나니 다음 문제가 보였다. 결과가 브라우저 localStorage에만 있다. 노트북을 바꾸면 세션이 없다.

그래서 앱에서 GitHub API로 세션 <span class="term" data-tip="문자열·숫자·불리언·배열·객체·null을 표현하는 텍스트 데이터 형식. 주석과 trailing comma는 표준 JSON에 포함되지 않는다.">JSON</span>을 직접 커밋하는 동기화를 붙였다. 서버 없이 GitHub 저장소가 백엔드가 된 셈이다. 바로 다음 날 한계에 부딪혔는데, GitHub Contents API는 1MB 넘는 파일을 그 방식으로 못 읽는다. 동화 원문이 쌓인 세션은 쉽게 1MB를 넘는다. Blob API로 폴백하는 코드를 넣어 해결했다.[^blob]

6월 28일에는 <span class="term" data-tip="오픈 웨이트 모델을 로컬 장비에서 내려받아 실행하고 API로 호출하게 해주는 도구. 외부 모델 API의 호출 요금은 없지만 장비·전력·운영 비용은 별도다.">Ollama</span>를 붙여 로컬 모델 평가(비용 0)도 되게 했다. 클라우드 프론티어 모델과 내 컴퓨터의 2B급 온디바이스 모델을 같은 잣대로 비교하는 게 목적이었다. 같은 날 14모델 실측을 돌렸는데 180/273쌍에서 크레딧이 떨어져 멈췄다. 체크포인트 이어하기가 이미 있어서 낭비는 없었다.

## 검증 자료를 남기는 쪽으로

이 기간의 앱 커밋들을 돌아보면 방향이 하나로 수렴한다. 판정 근거와 전송 프롬프트를 빠짐없이 기록해 승인된 검토자가 감사할 수 있게 하는 쪽이다. 다만 2026-07-20 점검 현재 little-bard 저장소는 비공개여서, 아래 GitHub 링크는 권한 없는 방문자에게 404로 응답한다. 공개 검증을 표방하려면 비식별 자료를 별도 공개해야 한다.

- 각 쌍을 어느 모델이 만들었는지, 심판별 판정과 근거 전문(정방향·역방향 원문 그대로)
- 생성과 심판에 실제로 전송되는 프롬프트 원문
- 앱 안의 코드·프롬프트 탭에 통계 엔진 코드 자체를 띄우고, 원리 탭에 설계 결정의 근거와 출처를 문서화

이유는 실용적이었다. 이 시스템은 멘토님께 "이 평가를 믿어도 되는가"를 검증받아야 했다. 결과 숫자만 보여주면 검증이 안 된다. 판정 근거 원문까지 열어놔야 "심판이 이상한 이유로 이겼다고 하네"를 잡아낼 수 있다. 실제로 이 투명성 덕에 나중에 심판 파싱 버그와 편향을 데이터로 잡게 된다.

## 이번 편의 교훈

앱을 따로 만든 이유는 사람 평가의 진입장벽이었다. 설치 없이 파일 하나로 여는 것 자체가 요구사항이었다. 첫 실측은 존재하지 않는 모델 slug로 실패했고, 비싼 호출 전에 싼 사전 점검을 두게 됐다. 같은 엔진을 두 언어로 유지하면 갈라진다는 것, 그 방어선이 등가성 테스트라는 것도 이때 배웠다.

4편은 스케줄러다. 모델이 13개면 쌍이 78개고, 50권이면 판정이 수만 콜이다. <span class="term" data-tip="모든 후보 쌍을 빠짐없이 비교하는 구성. N개 모델이면 N(N−1)/2쌍이라 비용이 제곱으로 는다. 가장 정보가 많지만 가장 비싸다.">완전그래프</span>를 다 돌려야 하는지를 실험으로 확인했다.
