[1편](/posts/llm-eval-pipeline-from-scratch-bradley-terry/)에서 파이썬 평가 파이프라인을 만들었다. 그런데 파이프라인을 커밋한 6월 18일, 같은 날 480줄짜리 HTML 파일을 하나 더 커밋했다.[^v1] 한 달 뒤 이 파일은 3,400줄이 됐고, 결국 프론티어 모델 13개의 실측 평가 전체를 브라우저 안에서 돌리는 엔진이 됐다.

왜 같은 걸 두 번 만들었나. 그리고 그 결정이 어떤 문제를 불렀나.

---

## 파이썬으로 안 되는 게 하나 있었다: 사람

이 평가 시스템에서 LLM 심판의 순위는 절반이다. 나머지 절반은 **사람 골든셋** — 같은 동화 쌍을 사람도 평가해서 심판과의 일치도(Cohen's κ)를 재야 "심판을 믿어도 된다"가 성립한다.

문제는 그 "사람"이 나 혼자가 아니라는 것. 멘토님, 팀원, 나중엔 학부모까지 평가에 참여할 수 있어야 했다. 그 사람들에게 "Python 3.9 설치하고 터미널에서 스크립트 돌려주세요"라고 할 수는 없다.

그래서 요구사항을 이렇게 정했고, 이게 곧 아키텍처가 됐다:

> **파일 1개. 빌드·서버·의존성 전혀 없음. 더블클릭하면 바로 열린다.**[^appreadme]

React도 빌드 도구도 안 썼다. 단일 `index.html`에 순수 JS로 UI와 통계 엔진(Bradley-Terry, 부트스트랩, κ)을 전부 넣었다. 카톡으로 파일 하나 보내면 상대방 브라우저에서 그대로 도는 것 — 배포가 "파일 전송"인 앱이다. API 키가 없어도 고정 시드 합성 데이터로 전체 흐름이 도는 **데모 모드**를 기본으로 깔아서, 처음 여는 사람이 아무 설정 없이 30초 안에 "아 이런 식으로 평가하는구나"를 체험하게 했다.

트레이드오프는 알고 시작했다. 단일 파일은 자랄수록 모놀리스가 된다. 실제로 지금 3,420줄인데, 이 결정의 비용은 나중에 꽤 비싸게 치른다(5편에서 다룬다).

완성형 기준의 전체 구조를 먼저 보여주고 시작하는 게 이해에 빠를 것 같다. 아래가 이 앱의 실측 경로 전부다 — 이 글과 이후 편들은 이 그림의 각 상자가 왜 그렇게 생겼는지에 대한 이야기다.

## 첫 실측: 시작하자마자 실패

6월 19일, 처음으로 실제 API(OpenRouter)를 붙여 실측을 돌렸다. 결과는 실패. 원인을 파보니 허무했다 — **모델 목록에 넣어둔 slug 6개가 OpenRouter에 존재하지 않는 이름이었다.**[^slugfix]

모델 카탈로그를 문서와 기억으로 작성했던 게 화근이었다. 존재하지 않는 모델명은 호출 시점에야 404로 드러나고, 그때는 이미 다른 모델들 호출에 돈을 쓴 뒤다.

수정하면서 두 가지를 앱에 박았다:

1. **사전 점검** — 실측 시작 전에 선택된 모든 모델에 1토큰짜리 테스트 호출을 날려 slug 유효성을 먼저 확인한다. 실패 목록을 한 번에 보여주고 시작 자체를 막는다.
2. **에러 상세화** — HTTP 상태별로 원인을 사람 말로 번역한다(401=키 오류, 402=크레딧 부족, 404=slug 없음, 429=요청 한도).

당연한 걸 왜 처음부터 안 했나 싶지만, "일단 돌려보자"로 시작한 코드는 원래 이렇다. 같은 날 양방향 swap(위치 편향 통제)도 앱에 구현했다.

## 이중 구현의 저주, 첫 등장

6월 23일에 찾은 버그가 하나 있다. 같은 판정 데이터인데 **앱과 파이썬 파이프라인의 순위가 미묘하게 달랐다.** 원인: swap 집계 방식이 서로 달랐다. 파이썬은 (쌍, 심판)별로 정방향·역방향 판정을 **평균**해 contest 하나로 만드는데, 앱은 두 판정을 각각 넣고 있었다.[^swapfix]

왜 평균이어야 하나. 정방향과 역방향을 **별개의 contest 두 개**로 넣으면, 위치 편향이 만든 "정방향 A승 + 역방향 B승"이 실력 신호처럼 우도에 두 번 들어간다. (쌍, 심판) 단위로 먼저 평균해 contest 하나(s ∈ {0, ¼, ½, ¾, 1})로 접어야 위치 효과가 집계 단위 안에서 상쇄된다. 파이썬은 처음부터 이 방식이었고 앱이 어긋나 있었던 것.

앱을 파이썬과 같은 방식으로 통일하고, 두 구현이 같은 입력에 같은 순위를 내는지 **등가성 테스트**를 만들었다 — 같은 judgments를 양쪽 엔진에 넣어 Arena 점수를 대조하는 방식이다. 이때 처음으로 "같은 엔진을 두 언어로 유지한다"는 게 공짜가 아니라는 걸 체감했다. 한쪽을 고치면 다른 쪽도 고쳐야 하고, 잊으면 조용히 갈라진다. 이 드리프트 문제는 이후로도 계속 따라온다 — 5편에서 심판 temperature가, 그다음엔 부트스트랩 방식이 같은 방식으로 갈라져 있었음이 드러난다.

## 사람 평가 기능의 진화

앱의 존재 이유가 사람 평가였으니, 이 부분이 가장 많이 진화했다. 전부 6월 23일~24일 이틀에 몰려 있다:

| 기능 | 왜 필요했나 |
|---|---|
| 다중 평가자 (rater별 누적·전환) | 한 노트북으로 여러 명이 돌아가며 평가하는 상황 |
| 팀 분산 평가 (HTML 배포 → 결과 파일 병합) | 각자 자기 컴퓨터에서 평가하고 jsonl로 합치기 |
| 평가자 신뢰도 종합 (Krippendorff α) | 2명은 Cohen's κ로 되지만 N명 일치도는 α가 필요[^alpha] |
| 부분 평가 (대표 표본만) | 쌍이 수백 개면 다 못 본다 — 모델×단어수 비율을 유지한 층화 표본 |
| 실측 비용·ETA·예산 상한 | 실측은 돈이라, 예산 넘으면 자동 중단 |

층화 표본은 난수 없이 결정론적으로 뽑는다. 같은 데이터에 같은 n이면 항상 같은 표본 — 평가자들이 같은 쌍을 봐야 평가자 간 일치도가 성립하기 때문이다.

상태 설계도 이때 골격이 잡혔다. 앱의 전 상태는 `STATE` 객체 하나로 모이고, 여기서 갈라지는 저장 단위가 곧 데이터 모델이다:

| 상태 | 담는 것 | 왜 분리했나 |
|---|---|---|
| `pairs[]` | 쌍별 동화 원문(EN/KO)·심판별 점수/근거 전문 | 분석·리포트의 원시 데이터 — 근거 전문 보존이 원칙 |
| `human` / `raters` | 현재 평가자 화면용 투표 / 평가자별 누적 투표 | 여러 명이 한 앱을 쓰니 "화면"과 "장부"를 분리 |
| `live` | 실측 진행 체크포인트(생성물·판정 캐시·은퇴 상태) | 중단 지점부터 재개하려면 완료분과 진행분이 달라야 함 |
| `usage` / `modelCost` | 콜·토큰·비용 누적, 모델별 편당 비용 | 예산 가드와 품질×비용 Pareto의 입력 |
| `reviewIds` | 층화 표본으로 고른 쌍 id | 평가 범위 자체도 상태 — 새로고침에도 유지돼야 함 |

이 분리가 뒤에서 두 번 목숨을 구한다. 실측이 중단돼도 `live`의 판정 캐시 덕에 완료분을 다시 호출하지 않았고(6/28 크레딧 소진 때 실전 검증), 투표가 화면에서 사라졌을 때 `raters` 장부에 살아있었다(7편).

## 첫 완주, 그리고 GitHub가 저장소가 된 날

6월 24일, 처음으로 실측을 완주했다: **8모델 × 5심판, 84/84쌍.**[^firstrun] 그런데 완주하니 새 문제가 보였다. 결과가 브라우저 localStorage에만 있다는 것. 노트북을 바꾸면 세션이 없다.

그래서 앱에서 GitHub API로 세션 JSON을 직접 커밋하는 동기화를 붙였다. 서버 없이 GitHub 저장소가 백엔드가 된 셈이다. 바로 다음 날 한계에 부딪혔는데 — GitHub Contents API는 1MB를 넘는 파일을 그 방식으로 못 읽는다. 동화 원문이 쌓인 세션은 쉽게 1MB를 넘는다. Blob API로 폴백하는 코드를 넣어 해결했다.[^blob]

6월 28일에는 Ollama를 붙여서 **로컬 모델 평가(비용 0)**도 되게 했다. 클라우드 프론티어 모델과 내 컴퓨터의 2B급 온디바이스 모델을 같은 잣대로 비교하는 게 목적이었다. 같은 날 14모델 실측을 돌렸는데 180/273쌍에서 크레딧이 떨어져 멈췄다 — 이때는 "체크포인트에서 이어하기"가 이미 있어서 낭비는 없었다.

## 숨기지 않는 쪽으로

이 기간 앱 커밋들을 돌아보면 방향이 하나로 수렴한다: **전부 공개.**

- 각 쌍을 어느 모델이 만들었는지, 심판별 판정과 **근거 전문**(정방향·역방향 원문 그대로)
- 생성·심판에 실제로 전송되는 **프롬프트 원문**
- 앱 안의 [코드·프롬프트] 탭에 통계 엔진 코드 자체를 띄우고, [원리] 탭에 모든 설계 결정의 근거와 출처를 문서화

이유는 실용적이었다. 이 시스템은 멘토님께 "이 평가를 믿어도 되는가"를 검증받아야 했다. 결과 숫자만 보여주면 검증이 불가능하다. 판정 근거 원문까지 열어놔야 "심판이 이상한 이유로 이겼다고 하네"를 잡아낼 수 있다. 실제로 이 투명성 덕분에 나중에 심판 파싱 버그와 편향을 데이터로 잡아내게 된다.

## 정리

- 앱을 따로 만든 이유는 **사람 평가의 진입장벽** — 설치 없이 파일 하나로 여는 것 자체가 요구사항이었다.
- 첫 실측은 존재하지 않는 모델 slug로 실패했다. **비싼 호출 전에 싼 사전 점검**을 배웠다.
- 같은 엔진을 두 언어(Python/JS)로 유지하면 반드시 갈라진다. 등가성 테스트가 방어선이다.
- GitHub를 백엔드 삼아 서버 없이 다기기 동기화를 해결했다(1MB 한계는 Blob API 폴백).
- 투명성(근거 전문·프롬프트·코드 공개)은 장식이 아니라 검증 가능성의 조건이었다.

다음 편: 모델이 13개면 쌍이 78개, 50권이면 판정이 수천 번이다. "완전그래프를 다 돌려야 하나?"라는 멘토님의 질문에서 시작된 스케줄러 연구 — Swiss 토너먼트, 은퇴(successive elimination), 그리고 오은퇴 0% 검증.
