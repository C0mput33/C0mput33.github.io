“모델에게 창작 프롬프트와 난이도 프롬프트를 따로 보내는가?”, “FK·Coleman–Liau·ARI 점수가 나쁘면 창작 순위도 내려가는가?”를 연달아 확인하면서 데이터 흐름을 다시 점검했다. 답은 둘 다 **아니다**. 다만 “난이도가 순위와 완전히 무관하다”라고만 말해도 정확하지 않다.

> **결론부터:** 모델마다 본선 동화는 **한 번의 통합 요청으로 한 편** 생성한다. 그 한 편을 LLM 심판이 비교해 만드는 창작 순위와, 코드가 가독성 공식을 적용하는 난이도 진단은 서로 다른 경로다. **측정된 FKGL·Coleman–Liau·ARI 값은 Bradley–Terry(BT)에 들어가지 않는다.** 반면 “7세·약 500L 독자에게 써라”라는 **요청 자체**는 생성과 심판 양쪽에 전달되므로, 심각한 독자 수준 불일치는 전체 Winner에 간접 영향을 줄 수 있다.

![통합 생성 요청에서 창작 Bradley-Terry 순위, 자동 가독성 진단, 선택형 levelTrack 반응성 프로브로 나뉘는 전체 흐름](/assets/img/posts/2026-07/creative-level-two-track.svg)
_그림 1. 본선 생성은 하나이고 평가가 갈라진다. 청록색·주황색 경로의 수치는 보라색 BT 순위 경로로 합류하지 않는다._

## 1. 모델이 받는 본선 생성 요청은 하나다

브라우저와 Python 모두 원래의 창작 요청 뒤에 목표 독자 정보를 붙인다. 실제 형태를 줄이면 이렇다.

```text
[주제 · 캐릭터 · 형식 · 요청 단어 수]
Target reader age: 7.
Target reading level: approximately 500L Lexile;
match vocabulary and sentence complexity to that level.
```

따라서 “재미있는 동화 한 편”과 “쉬운 동화 한 편”을 따로 만들고 어느 쪽을 쓸지 고르는 구조가 아니다. 한 동화가 주제·형식·길이와 독자 수준을 **동시에** 만족해야 한다. 이를 순수 문학성 순위라고 부르기보다는 **목표 독자 조건부 창작 품질(target-conditioned creative quality)** 순위라고 부르는 편이 정확하다.

실측 API 호출 수와 생성물 수는 구분할 필요가 있다. 본선 후보 모델은 프롬프트마다 영어 동화 한 편만 만든다. 유효한 원문에는 사람 검토용 한국어 번역 호출이 한 번 더 붙지만, 번역은 후보 생성도 아니고 판정 입력도 아니다. 순위는 영어 원문만으로 계산한다.

## 2. 창작 순위는 LLM의 전체 Winner 한 줄로만 만든다

같은 프롬프트에서 나온 두 모델의 동화를 익명 A/B로 놓고 다계열 LLM jury가 비교한다. 심판은 다음 정보를 함께 받는다.

- 원래 통합 생성 요청
- 목표 연령과 목표 Lexile
- 코드가 센 두 원문의 실제 영어 단어 수
- 동일한 4,000자 상한을 적용한 Story A와 Story B

각 심판에게 A/B와 B/A 순서를 모두 물어 위치 편향을 줄이고, 같은 회사 계열 모델의 자기평가는 제외한다. 심판은 안전을 하드 게이트로 보고, 그다음 핵심 요청·독자 대상 준수, 요청 형식에 맞는 서사 완성도, 재미·자연스러움·독창성 순으로 전체 선호를 결정한다. 숫자 가중합은 쓰지 않는다.

심판이 내는 축별 A/B/Tie는 “왜 이겼는가”를 분해하는 진단이다. **BT에 들어가는 것은 각 심판의 정·역방향 `WINNER`를 접은 contest뿐**이다. 엄격한 Winner 파싱에 실패하면 의미상 무승부 0.5로 바꾸지 않고 `invalid`로 제외하며, 비용 보호를 위해 기본 재호출도 하지 않는다.

## 3. FKGL·Coleman–Liau·ARI는 같은 동화를 재는 별도 자다

본선 영어 원문은 동시에 자동 진단 경로로도 간다. 여기에는 LLM 심판이나 추가 생성이 없다.

1. Flesch–Kincaid Grade Level(FKGL): 문장당 단어 수와 단어당 음절 수
2. Coleman–Liau Index: 100단어당 글자 수와 문장 수
3. Automated Readability Index(ARI): 단어당 글자 수와 문장당 단어 수

세 식의 추정 학년을 평균해 단일식의 특이성을 조금 완화하고, 이를 학년 밴드로 보간한 **Lexile 근사값**으로 표시한다. 여기서 중요한 단어는 “근사”다. 이 값은 MetaMetrics의 공인 Lexile 측정치가 아니며, 짧은 동화에서는 희귀한 긴 단어 하나나 문장부호 하나에도 크게 흔들릴 수 있다.[^lexile]

그래서 이 경로는 다음을 보여주는 진단 카드로만 쓴다.

- 목표 근처에 들어온 비율(Match Rate ±80L)
- 목표와의 평균 절대 편차(MAE)
- 같은 요청을 반복했을 때의 난이도 표준편차
- 요청한 단어 수 구간 준수율

이 카드의 숫자는 BT 적합, Arena 점수, retire2 은퇴 판정에 들어가지 않는다.

## 4. 그렇다면 Lexile은 순위에 영향을 주는가?

**요청한 목표값은 영향을 줄 수 있고, 사후 측정값은 영향을 주지 않는다.** 둘을 같은 “Lexile”이라고 부르면 혼동이 생긴다.

| 신호 | 누가 만드는가 | BT에 들어가는가 |
|---|---|---|
| 목표 연령·목표 Lexile 문구 | 평가 설정 | 심판의 요청·독자 적합 판단을 거쳐 **간접 반영** |
| 실제 영어 단어 수 | 코드 | 심판의 길이 준수 판단을 거쳐 **간접 반영** |
| 전체 `WINNER` | LLM 심판 | **직접 반영** |
| 축별 A/B/Tie | LLM 심판 | 아니오, 진단 전용 |
| FKGL·Coleman–Liau·ARI 합성값 | 코드 | 아니오 |
| Lexile 근사 Match Rate·MAE | 코드 | 아니오 |
| 한국어 번역 | 번역 모델 | 아니오, 표시 전용 |
| levelTrack 반응성 β | 코드 | 아니오 |

예를 들어 자동 측정값이 목표 500L에서 150L 벗어났다는 이유만으로 BT 점수를 차감하는 식은 없다. 하지만 한 모델이 7세 독자에게 지나치게 추상적이고 복잡한 문장을 쓰고 다른 모델은 자연스럽게 맞췄다면, 심판이 원 요청 준수와 연령 적합성에서 후자를 고를 수 있다. 제품 요구를 반영한 의도된 경로다.

문학성만 완전히 고립한 리더보드가 필요하다면 독자 수준 문구를 심판에서 제거한 별도 실험을 만들어야 한다. 현재 본선과 섞으면 평가 질문 자체가 달라진다.

## 5. `levelTrackOn`은 켰을 때만 동화를 더 만든다

본선 한 편의 절대 난이도만 재면 “우연히 쉬웠는가”와 “난이도 지시를 잘 따르는가”를 구분하기 어렵다. 선택형 `levelTrackOn`은 후자를 보기 위한 민감도 실험이다.

- **OFF:** 추가 생성 0회
- **ON:** 선택한 프로브마다 목표를 −1학년·기준·+1학년으로 바꿔 모델당 3편 추가 생성
- 각 출력의 합성 측정 학년을 목표 학년에 회귀해 반응성 기울기 β 계산
- β=1이면 목표를 한 학년 바꿀 때 측정 난이도도 평균 한 학년 움직였다는 뜻
- 번역과 LLM 심판 없음, BT와 은퇴 판정에 영향 없음

추가 생성 호출 수는 대략 `대상 모델 수 × 프로브 수 × 3`이다. retire2라면 트랙 실행 시점의 생존 모델만 대상으로 한다. 이 옵션은 비용을 실제로 늘리므로 기본 본선과 구분해 켜야 한다.

## 6. 실제 비용 구조를 한 줄로 쓰면

프롬프트 수를 P, 후보 모델 수를 N, 유효 심판 수를 J라고 할 때 full 스케줄의 큰 항은 다음과 같다.

```text
본선 후보 생성       P × N
사람 표시용 번역     P × N                 (유효 원문만)
페어와이즈 판정      P × C(N,2) × J × 2    (정·역방향)
levelTrack 옵션      N' × probe × 3         (ON일 때만)
```

판정이 비용 대부분을 차지하므로 retire2의 쌍-군집 CI와 은퇴 동작은 그대로 유지했다. 프롬프트-군집 CI는 저장된 contest를 다시 계산하는 **보고용 참고 구간**으로 병기한다. 이 계산은 API 호출을 늘리지 않으며, 독립 프롬프트가 10개 미만이면 “불안정”으로 표시한다. 다만 retire2의 관측 결과에 따른 은퇴 경로 자체를 다시 시뮬레이션하지 않으므로 두 CI 모두 그 관측 경로에 조건부다.

## 7. 이 분리가 해결하는 것과 해결하지 못하는 것

해결하는 것은 명확하다. 가독성 공식의 가짜 정밀도가 창작 순위를 기계적으로 오염시키지 않고, 창작 품질·요청 준수·자동 난이도·난이도 반응성을 각각 추적할 수 있다. 옵션을 끄면 프로브 비용도 0이다.

하지만 자동 프록시를 분리했다고 해서 본선이 “독자 수준과 무관한 순수 창작 대회”가 되는 것은 아니다. 목표 독자는 생성 요청과 심판 맥락에 실제로 들어간다. 또한 142편의 고유 동화가 여러 모델쌍과 심판 판정에 반복되는 의존성, 적응형 은퇴가 만드는 선택 효과는 별도의 통계 문제다. 프롬프트-군집 보고용 CI와 조건부성 경고는 그 한계를 드러내는 장치이지, 적은 독립 프롬프트를 갑자기 큰 표본으로 바꾸는 마법은 아니다.

정리하면 이 시스템은 **한 번 생성하고, 두 방식으로 본다.** LLM jury는 목표 독자에게 더 좋은 이야기인지 상대 비교하고, 결정론적 코드는 그 이야기가 표면적으로 얼마나 쉽고 지시 변화에 얼마나 반응하는지 잰다. 두 결과는 나란히 보여주되 숫자를 몰래 섞지 않는다.

[^lexile]: Stenner et al., [The Lexile Framework for Reading Technical Report](https://metametricsinc.com/wp-content/uploads/2017/07/Stenner_Burdick_Sanford__Burdick-_The_LFR_Technical_Report.pdf). Lexile은 어휘·문장 난이도를 결합한 별도 프레임워크이며 짧은 텍스트일수록 측정 표준오차가 커진다. 이 프로젝트의 3식 합성·학년 보간값은 공인 Lexile이 아니다.
