7월 14일, 드디어 진짜 돈을 썼다. 프론티어 13개 모델(GPT-5.2/5.5, Claude Opus 4.8/4.7, Gemini 3.1 Pro, Kimi K2.6, Qwen 3종, Gemma 2종 등)에 동화 50권 평가를 걸었다. [3편](/posts/llm-eval-scheduler-retire2-successive-elimination/)에서 검증한 retire2 스케줄러, 3계열 심판 jury(GPT-5.4 · Claude Sonnet 4.6 · Gemini 2.5 Pro), 양방향 swap — 준비한 걸 전부 켰다.

몇 시간 뒤 로그가 이렇게 끝나 있었다.

```
⚠ 중단: Unexpected end of JSON input
```

50권 중 13권. 797쌍, API 5,070콜, 토큰 8.03M, **$46.76을 쓴 뒤**였다.[^run]

## 그래도 retire2는 일했다

중단 얘기 전에, 스케줄러부터. 완주했다면 full 완전그래프는 3,900쌍이었다. retire2는 배치 2(10권)에서 **9개 모델을 은퇴**시키고 생존 4개(GPT-5.2, GPT-5.5, Kimi, GLM)만 남겼다. 은퇴 판정은 시뮬레이션에서 정한 그대로 "CI 분리 2연속"이 수행했고, 은퇴 모델들은 은퇴 시점의 점수·CI로 리더보드에 남았다.

중단 시점 기준으로 13개 모델의 순위와 95% CI가 이미 확정돼 있었다. 남은 37권이 할 일은 상위 4개끼리의 순서를 다듬는 것뿐이었어서, 중단에도 불구하고 **13모델 리더보드 자체는 완결된 결과로 쓸 수 있었다.** 전체 표를 남긴다:

복원 검증도 했다. 저장된 전 판정(797쌍 × 심판 3 = 2,391 contest)으로 BT를 다시 적합해 발행된 Arena 점수와 대조 — 일치. 파싱·계산에 오염이 없다는 확인이다.

## 중단의 진짜 원인은 따로 있었다

"Unexpected end of JSON input"은 심판 API 응답 하나가 깨져서 난 파싱 오류다. 그런데 이건 방아쇠일 뿐, 진짜 문제는 **이런 사고가 나면 안 되는 상태로 몇 시간을 달리고 있었다**는 것이다.

앱은 진행 상태를 1.5초마다 localStorage에 저장한다(2편의 "이어하기"). 그런데 localStorage에는 약 10MB 한도가 있고, 동화 원문 + 심판 근거 전문을 다 담는 이 실측은 쌍당 약 13KB — **775쌍 근처에서 한도를 넘는다.** 저장 코드는 이랬다:

```javascript
function saveState(){
  try{ localStorage.setItem("lb_eval_v1", JSON.stringify({...STATE, apiKey:""})); }
  catch(e){}   // ← 용량 초과가 여기서 조용히 삼켜졌다
}
```

`catch(e){}`. 저장 실패가 **아무 소리 없이** 무시됐다. 체크포인트는 38쌍 시점에 얼어붙어 있었고, 이후 $45어치의 판정은 **탭 메모리에만** 존재했다. 중단됐을 때 탭이 살아있었기에 망정이지, 브라우저가 죽었다면 전부 날렸다.

사고 후 조치: 저장 실패 시 경고를 로그에 띄우고, 10쌍마다 IndexedDB(용량 무제한)+GitHub에 자동 백업하고, 진행 전체를 파일로 내리는 체크포인트 버튼을 달았다. 교훈은 한 줄이다 — **빈 catch 블록은 미래의 나에게 지뢰다.** 특히 그 안에서 돈이 움직이고 있다면.

## 발견 1: 심판은 자기 식구에게 후했다 (t = 6.43)

이 런에는 부끄러운 설정이 하나 있었다. 편향 통제 중 하나인 **자기 계열 심판 제외가 꺼진 채**(allowSelfEval=true) 수집됐다. 덕분에(?) 자기 계열 편향을 전 쌍에서 정량 측정할 수 있었다.

방법을 구체적으로 적어둔다. 그냥 "자기 계열 쌍의 평균 점수"를 보면 안 된다 — 자기 계열 모델이 실제로 더 잘 썼을 수도 있으니까. 그래서 **동료 심판을 대조군**으로 썼다: 같은 쌍을 본 세 심판 중 한 심판의 점수에서 나머지 두 심판의 평균을 뺀 값(peer-relative Δ)을 만들면, 쌍의 실제 품질 차는 소거되고 그 심판의 상대적 성향만 남는다. 이 Δ가 "자기 계열이 낀 쌍"에서 체계적으로 +인지를 t-검정했다.

| 심판 (계열) | 자기편애 Δ | t |
|---|---|---|
| GPT-5.4 (openai) | +0.029 | +1.58 |
| Claude Sonnet 4.6 (anthropic) | +0.056 | +2.95 |
| Gemini 2.5 Pro (google) | **+0.095** | **+5.97** |
| **3심판 합산** | **+0.066 (≈ +15 Arena pt)** | **+6.43** |

**세 심판 모두 자기 계열에 +**였고, 합산 t=6.43은 우연으로 볼 수 없는 수준이다. Gemini가 가장 셌는데, 후보 중 구글 계열이 4개로 가장 많아 표본·영향이 컸다. 다계열 jury 평균이 이걸 부분 상쇄하지만 계열별 후보 수가 불균형이라 완전 상쇄는 아니다. 결론: **인접 순위(±15pt 이내)는 오차로 간주하고, 다음 런은 자기평가 제외를 반드시 켠다.** 1편에서 "구조로 막는다"고 했던 편향이 실제로, 측정 가능한 크기로 존재했다.

## 발견 2: BT와 Elo가 다른 순위를 냈다 (Spearman 0.742)

앱은 주 순위(BT)와 보조 지표(순차 Elo)를 함께 보여주는데, 이 런에서 둘이 눈에 띄게 갈렸다. 극적인 건 Claude 두 모델의 역전이다:

| 모델 | BT(Arena) 순위 | Elo 순위 |
|---|---|---|
| Claude Opus 4.8 | 10위 | **4위 (▲6)** |
| Claude Opus 4.7 | **4위** | 9위 (▼5) |

원인은 retire2와 Elo의 상성이다. Elo는 전원 1000에서 시작해 경기마다 `R ← R + K·(s − E)`로 갱신되는 온라인 지표다(기대승률 `E = 1/(1+10^((R_상대−R)/400))`, K=32). 순서대로 갱신되기 때문에, 배치 2에서 은퇴한 모델의 Elo는 **그 시점에 얼어붙고** 생존 모델만 계속 갱신된다. 얼어붙은 값과 갱신되는 값을 한 표에 놓고 비교하는 셈이다. 게다가 K-팩터 갱신은 "그 시점 상대 레이팅"에 민감해서, 유리한 타이밍의 승리가 과대 반영된다. BT는 모델별 경기 수 불균형과 대전 시점을 MLE로 올바르게 처리하므로 이 왜곡이 없다.

1편에서 "오프라인 고정 셋엔 BT"라고 이론으로 골랐던 게, 실측에서 **7계단 역전**이라는 실물로 확인됐다. Elo 이탈은 "데이터가 작고 retire로 구조화됐다"는 진단 신호이지, Elo가 뭔가를 더 발견한 게 아니다.

## 발견 3: 신형이 구형보다 낮다

리더보드에서 가장 눈에 걸린 것: **Opus 4.8(953, CI 923–977)이 Opus 4.7(1052, CI 1023–1077)보다 유의하게 낮았다.** CI가 겹치지 않는다.

자기 계열 편향으로는 설명이 안 된다 — 그 편향은 두 Opus를 똑같이 밀어주기 때문이다. Opus끼리의 직접 대결에선 Anthropic 심판이 양쪽을 같은 계열로 보니 상대적 편향도 없다. 즉 GPT·Gemini를 포함한 중립 판정이 아동 동화라는 이 도메인에서 신형을 낮게 본 것이다. "최신 모델이 항상 낫다"가 도메인 특화 평가에선 성립하지 않을 수 있다는, 이 시스템을 만든 보람이 있는 신호였다. 도메인 특성인지 모델 회귀인지는 분리 실험이 필요해서 후속 과제로 남겼다.

## 덤: 지표가 전부 0인데 문법은 멀쩡했다

결과 화면의 난이도 적합도(Match Rate·MAE)가 전 모델 0으로 나오는 버그도 이날 잡았다. 원인이 어이없다 — 집계 함수 한 줄 **중간에** `//` 주석이 박혀서, 그 뒤의 누적문 세 개가 통째로 주석 처리돼 있었다. 문법상 유효한 코드라 구문 검사는 통과한다. 죽은 코드는 에러를 안 낸다.[^bugfix]

수정 후 재계산하니 MAE 380L, Match Rate 2.9%가 나왔다. 낮은 건 버그가 아니라 측정 도구의 한계다 — 측정 Lexile 중앙값이 868L로 목표(~500L)보다 계통적으로 높게 나오는, 짧은 글에서 가독성 공식이 과대추정하는 문제다. 이 얘기는 5편에서 제대로 다룬다.

## 정리

- retire2는 실전에서 설계대로 작동했다(9모델 조기 은퇴, 중단에도 13모델 CI 확정).
- 중단의 방아쇠는 API 응답이었지만 **원인은 빈 catch 블록**이었다. 돈이 걸린 루프의 저장 실패는 침묵하면 안 된다.
- 자기 계열 편향은 실재하고 측정 가능했다(합산 t=6.43). 편향 통제는 장식이 아니다.
- BT vs Elo 역전(Spearman 0.742)으로 "왜 BT인가"가 실측으로 증명됐다.
- 신형 < 구형(Opus 4.8 < 4.7, CI 비겹침) — 도메인 특화 평가가 존재하는 이유.

다음 편: 이 실측 결과를 들고 AI 에이전트에게 엔진 전체의 교차 리뷰를 시켰다. 돌아온 건 결함 5개 — 그중엔 이 리더보드의 신뢰구간을 실제보다 좁게 만들고 있던 통계 버그도 있었다.
