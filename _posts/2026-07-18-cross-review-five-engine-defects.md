## 결함 1: 심판 temperature가 0.7이었다

API 호출 함수에 <span class="term" data-tip="다음 토큰 확률분포의 평탄함을 조절하는 값. 낮을수록 높은 확률의 토큰에 집중하지만 temperature 0도 제공자·모델·실행 환경에 따라 완전한 재현성을 보장하지 않는다.">temperature</span>가 하드코딩돼 있었다.

```javascript
// 수정 전: 생성·번역·판정이 전부 이 함수 → 전부 temperature 0.7
body: JSON.stringify({model, messages, temperature: 0.7, ...})
```

동화 생성에는 다양성을 위해 0.7을 선택할 수 있다. 판정까지 같은 값으로 나가면서 반복 호출의 샘플링 변동이 커진 것이 문제였다. 파이썬 파이프라인의 심판은 처음부터 `temperature: 0.0`이어서 두 구현도 달랐다. temperature 0은 변동을 줄이지만 제공자와 모델 구현까지 고정하지 않으므로 완전한 재현성을 보장하지는 않는다.

수정은 한 줄 개념이다. 호출에 옵션을 열고, 판정 두 콜(정방향·역방향)만 temp 0으로 보냈다.

```javascript
const r1 = await orCall(judge, SYS, user(A,B), {temp: 0});  // 판정 샘플링 변동 축소
const r2 = await orCall(judge, SYS, user(B,A), {temp: 0});  // 생성·번역은 별도 값 유지
```

## 결함 2: 신뢰구간이 실제보다 좁게 계산됐다

리더보드의 95% CI를 만드는 <span class="term" data-tip="가진 데이터에서 복원추출로 여러 번 가짜 표본을 만들어 같은 계산을 반복하고, 그 결과들의 흩어짐으로 추정치의 불확실성을 재는 방법. 표본이 모집단을 대표한다면 재표집의 흔들림이 실제 추정 오차와 비슷하다는 원리다.">부트스트랩</span>에 문제가 둘 겹쳐 있었다.

반복이 60회였다. 데이터가 클수록 반복을 줄이는 코드(`n>1500 ? 60 : ...`)라, 정작 797쌍 실측에서 60회만 재표집했다. 반복 수가 좁히는 건 데이터 노이즈가 아니라 <span class="term" data-tip="데이터를 크기순으로 늘어놓았을 때 아래에서 몇 % 위치에 있는 값. 2.5백분위와 97.5백분위 사이가 가운데 95%를 담는 구간이 된다.">백분위</span> 끝값 자체의 <span class="term" data-tip="난수를 이용한 반복 시뮬레이션으로 답을 근사하는 계산 방법. 반복이 많을수록 정확해지고, 적으면 결과 자체가 흔들린다. 부트스트랩 반복 횟수가 중요한 이유다.">몬테카를로</span> 노이즈라서, 발상이 뒤집혀 있었다. 60개 표본에서 2.5백분위는 사실상 아래에서 1~2번째 값인데, 극단 <span class="term" data-tip="데이터를 정렬했을 때 k번째에 오는 값. 중앙값은 안정적이지만 최솟값이나 최댓값 같은 극단 쪽 값은 표본이 적을수록 심하게 흔들린다. 신뢰구간의 끝점이 바로 이 극단 쪽 값이다.">순서통계량</span>은 표본이 적을수록 심하게 흔들린다. 끝점이 흔들리는 <span class="term" data-tip="같은 절차로 표본을 반복 수집할 때 정해진 비율만큼 모수를 포함하도록 만든 구간. 두 개의 95% 신뢰구간이 겹치는지만으로 차이의 유의성을 판정할 수는 없다.">신뢰구간</span>이었던 셈이다.

재표집 단위도 틀렸다. 판정(contest) 하나하나를 독립으로 뽑았는데, 같은 쌍을 본 심판 3명의 판정은 서로 상관돼 있다. 상관된 것을 독립 취급하면 분산을 과소평가하고, CI가 실제보다 좁아진다. 이 CI가 <span class="term" data-tip="이 프로젝트의 은퇴 스케줄러. 배치(5권)마다 BT와 CI를 갱신하고, CI 분리가 2연속 확인된 모델만 은퇴시킨다. 한 번의 우연으로 탈락시키지 않는 확인 절차가 이름의 유래다.">retire2</span>의 은퇴 판정을 직접 구동하니, 좁은 CI는 과신이고 과신은 억울한 조기 은퇴로 이어질 수 있다.

당시 앱 수정은 쌍(pair) 단위 클러스터 재표집과 1000회 고정이었다. 쌍을 뽑으면 그 쌍의 심판 판정 전부가 함께 들어간다. 이는 contest 독립 가정보다 낫지만 끝이 아니다. 같은 프롬프트와 같은 동화가 여러 모델쌍에 반복되므로 쌍들 사이에도 의존성이 남고, 프롬프트는 13개뿐이다.

```javascript
for(let i=0;i<N;i++){
  const p = pairs[rand()*N | 0];               // 쌍 단위 복원추출
  for(const jd in p.juryScore)                  // 그 쌍의 심판 판정을 '함께'
    res.push({a:p.a.model, b:p.b.model, s:p.juryScore[jd]});
}
```

이 수정이 실측 리더보드를 어떻게 바꿨는지는 7편에 있다. 순위는 안 바뀌었고, 쌍-군집 CI가 평균 51% 넓어졌다. 다만 그 구간도 프롬프트·동화 재사용과 적응형 은퇴 경로를 완전히 반영하지 않으므로 최종 확증 CI로 읽으면 안 된다.

## 결함 3: 파싱 실패가 무승부로 집계됐다

심판 응답에서 승자를 뽑는 파서는 `Winner: A` 형식을 정규식으로 잡는다. 못 잡으면 tie(0.5)로 처리된다. 예컨대 `Winner is A`라고 쓰면 매치가 안 되고, 그 판정은 아무 흔적 없이 무승부로 집계에 들어간다. 파이썬 쪽은 파싱 성공 여부(parse_ok)를 기록하는데 JS 쪽엔 그 텔레메트리가 없었다. temperature 0.7(결함 1)로 형식 이탈이 잦았다면 결과가 0.5 쪽으로 끌려가고 있었을 수 있다.

정규식을 넓히고 싶은 유혹이 있었지만 일부러 안 넓혔다. 과거에 `\b` 없는 느슨한 정규식이 본문의 "the winner: **a** close call"에서 'a'를 A 승리로 오파싱하던 버그가 실제로 있었다. 느슨한 파서는 침묵 오염을 다른 침묵 오염으로 바꿀 뿐이다.

대신 실패를 보이게 만들었다. 파싱 성공과 실패를 집계해 실패 시 로그에 즉시 경고하고, 결과 화면 체크리스트에 파싱 성공률을 상시 노출한다. 고칠 수 없으면 최소한 셀 수는 있어야 한다.

## 결함 4: 빈 catch 블록

5편 중단 사고의 원인이던 <span class="term" data-tip="브라우저가 사이트마다 제공하는 간단한 키-값 저장 공간. 코드 한 줄로 쓰고 읽지만 용량이 약 10MB로 제한되고, 초과하면 저장이 예외로 실패한다.">localStorage</span> 침묵 실패를 이번에 근본 수정했다. 용량을 실측해보니 쌍당 약 13KB였다. 동화 3편이 4KB쯤, 심판 3명 × 양방향 근거 전문이 9KB쯤이다. 10MB 한도면 775쌍 근처가 임계점이고, 실제 797쌍 <span class="term" data-tip="진행 상태를 통째로 저장해둔 지점. 중단되거나 크레딧이 떨어져도 완료분을 다시 호출하지 않고 그 지점부터 이어서 실행할 수 있다.">체크포인트</span> 파일이 10.02MB였다. 5편의 중단은 우연이 아니라 산수였다.

수정: 저장 실패 시 <span class="term" data-tip="브라우저에 내장된 본격 데이터베이스. localStorage와 달리 용량이 사실상 무제한이고 비동기로 동작해 큰 데이터 보관에 맞다.">IndexedDB</span>(사실상 무제한)로 즉시 전량 폴백하고, 앱을 켤 때 IndexedDB 쪽이 더 완전하면 자동 복구해 이어하기 지점을 잡는다. localStorage(빠름), IndexedDB(무제한), GitHub(원격)의 3중이 됐다.

## 결함 5: 난이도 절대값의 낮은 신뢰도

5편 끝에 "<span class="term" data-tip="목표 난이도의 허용 범위(±밴드) 안에 들어온 비율. 절대 눈금이 틀린 측정기로 재면 실력과 무관하게 낮게 나올 수 있다.">Match Rate</span> 2.9%는 도구의 한계"라고 적었다. 그 한계의 정체는 이렇다. 텍스트 난이도를 가독성 공식(Flesch-Kincaid, Coleman-Liau, ARI 합성)으로 재고 학년→<span class="term" data-tip="영어 텍스트의 읽기 난이도를 나타내는 지수(대략 0~1800L). 단어 빈도와 문장 길이를 회귀로 결합해 만들며, 독자 수준과 텍스트 수준이 같으면 약 75% 이해율을 예측한다.">Lexile</span>로 보간하는데, 이 공식들은 원래 긴 지문용이다. 문장 몇 개짜리 동화에선 희귀 장음절어 하나에 학년이 튀고, 공인 보정(MetaMetrics) 없는 매핑이라 절대값이 계통적으로 높게 나온다. 목표 500L짜리 동화를 재면 중앙값 868L이 나오는 이유다. 짧은 글의 불안정성은 Lexile을 만든 쪽도 정량으로 인정한다. 기술보고서에 따르면 약 125단어 슬라이스 하나의 표준오차가 178L이고, 4슬라이스짜리 책도 ±89L인데, 전쟁과 평화처럼 4,082슬라이스면 ±3L로 떨어진다.[^lexile] 우리 동화는 30~500단어라 애초에 오차가 큰 구간에서 재고 있는 것이다.

전면 재설계 대신 두 가지로 정리했다. 이상치가 평균과 분산을 왜곡하지 않게 합성 학년을 [0,14]로 클램프했고, 절대값 대신 <span class="term" data-tip="목표 레벨을 한 단계 바꿨을 때 측정 난이도가 실제로 몇 단계 움직였는지의 회귀 기울기. 1.0이면 지시한 만큼 정확히 반응한다는 뜻. 절대 눈금이 틀린 자로도 변화량은 잴 수 있다는 발상이다.">반응성 β</span>를 주 지표로 삼았다. β는 "레벨을 한 단계 낮춰달라고 하면 실제로 한 단계 쉬워지는가"의 회귀 기울기다. 절대 눈금이 틀린 자로도 변화량은 잴 수 있다. 실측에서 β는 전 모델 양수(+1.8~+3.2)로, 모델들이 난이도 지시에 실제로 반응한다는 것까지는 확인됐다.

## 수정 과정에서 다시 나온 드리프트

결함 2를 고치면서 파이썬 쪽 부트스트랩도 쌍 클러스터로 맞추려 했다. 그러자 retire 스케줄러의 회귀 테스트가 깨졌다. 파이썬 retire의 결정적 테스트들(어느 배치에서 누가 은퇴하는지까지 고정)이 기존 <span class="term" data-tip="집계의 최소 단위. 한 쌍에 대한 한 심판의 판정 하나(양방향 평균을 마친 점수)를 말한다. 심판 셋이 본 쌍이면 contest 3개가 BT에 들어간다.">contest</span> 단위 CI에 보정돼 있었던 것이다. CI 방식을 바꾸면 은퇴 시나리오가 통째로 바뀐다.

당시 파이썬 수정은 원복했고 앱(JS)만 쌍-군집으로 개선했다. 2026-07-20 보정에서는 **은퇴 판정 CI를 건드리지 않은 채** Python과 앱의 공식 운영 구간을 쌍-군집으로 맞췄다. 프롬프트-군집 구간은 API 호출 없이 저장 판정으로 계산하는 보고용 지표로만 추가했고, 프롬프트 10개 미만에는 불안정 경고를 붙였다. 은퇴 판정 방식을 바꾸려면 test_schedule 재보정과 50프롬프트 full 검증을 같이 해야 한다는 제약은 그대로다.

## 검증

다섯 수정 전부에 안전망을 붙였고, 재현 가능하게 남겼다.

- 앱 전체 JS 구문 검증. `<script>` 블록을 추출해 Node의 `vm.Script`로 파싱한다. 3,400줄 단일 파일에선 오타 하나가 앱 전체를 백지로 만들기 때문에 편집마다 도는 1차 관문이다.
- 바뀐 순수 함수의 단위 테스트 13종. 파서는 과거 오파싱 사례들("Winner: Although...", "winner: a close call", `**A**` 볼드)을 회귀 케이스로 고정했고, 부트스트랩은 시드 고정 결정성과 쌍 단위 재표집, 빈 입력을 검사한다.
- 파이썬 테스트 확인. `python3 eval/tests/test_eval.py`(17개)와 `test_schedule.py`(6개) 전부 통과. 파이썬을 안 건드렸다는 사실 자체를 테스트로 확인했다.

| 3 | 파싱 실패 = 조용한 tie | 침묵 오염, 텔레메트리 부재 | 실패 집계·경고·노출 (정규식은 안 넓힘) |
| 4 | localStorage 빈 catch | 침묵 데이터 유실 | IndexedDB 폴백 + 자동 복구 |
| 5 | 절대 Lexile 미보정 | 도구의 계통 오차를 성능으로 오독 | 클램프 + 반응성 β 중심 |

다섯 개 전부 오류 없이 잘못된 값을 돌려주는 버그였다. 크래시는 바로 드러나지만, 좁은 CI·무승부·0점을 정상 출력처럼 돌려주는 코드는 만든 사람 눈에 잘 띄지 않는다. 심판에게 자기평가를 금지시킨 논리를 코드 리뷰에도 적용했어야 했는데, 늦게라도 하길 잘했다.

7편은 결함 2를 고친 뒤 신뢰구간을 다시 계산한 기록이다.
