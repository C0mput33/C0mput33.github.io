[4편](/posts/46-dollar-live-eval-crash-lessons/)의 실측 리포트를 정리하다가 어떤 모순을 깨달았다. 이 시스템은 심판 LLM이 **자기 계열 모델을 평가하지 못하게** 설계돼 있다. 자기 것에 후한 편향이 실측에서 t=6.43으로 검출되기도 했다. 그런데 그 시스템의 **코드 자체는** 줄곧 만든 사람(과 만든 에이전트)만 들여다보고 있었다.

그래서 7월 15일, 실측 데이터를 쥔 채로 별도의 AI 에이전트에게 엔진 전체 감사를 맡겼다. 주문은 하나 — "주석과 문서를 믿지 말고 코드만 봐라. 칭찬 말고 결함을 찾아라."

돌아온 지적 중 검증을 통과한 게 5개였다. 하나씩 코드로 재확인하고, 고치고, 테스트를 붙였다.[^commit]

## 결함 1 — 심판이 매번 다른 기분으로 채점하고 있었다

API 호출 함수에 temperature가 하드코딩돼 있었다.

```javascript
// 수정 전: 생성·번역·판정이 전부 이 함수 → 전부 temperature 0.7
body: JSON.stringify({model, messages, temperature: 0.7, ...})
```

동화 **생성**은 0.7이 맞다. 창작이니까. 문제는 **판정**까지 같은 함수를 타면서 0.7로 나갔다는 것. 같은 두 동화를 두 번 물으면 다른 승자가 나올 수 있는, 비결정적 심판이었다. 더 아픈 건 파이썬 파이프라인의 심판은 처음부터 `temperature: 0.0`이었다는 것 — 2편에서 말한 **이중 구현 드리프트**가 또 있었다.

수정은 한 줄 개념이다. 호출에 옵션을 열고, 판정 두 콜(정방향·역방향)만 `{temp: 0}`으로:

```javascript
const r1 = await orCall(judge, SYS, user(A,B), {temp: 0});  // 판정 = 결정적
const r2 = await orCall(judge, SYS, user(B,A), {temp: 0});  // 생성·번역은 0.7 유지
```

## 결함 2 — 신뢰구간이 실제보다 좁게 나오고 있었다

리더보드의 95% CI를 만드는 부트스트랩에 문제가 둘 겹쳐 있었다.

**반복이 60회.** 데이터가 클수록 반복을 줄이는 코드(`n>1500 ? 60 : ...`)라, 정작 797쌍 실측에서 60회만 재표집했다. 2.5/97.5 백분위의 끝값은 60개 표본으로는 몬테카를로 노이즈가 크다.

**재표집 단위가 틀렸다.** 판정(contest) 하나하나를 독립으로 뽑았는데, 같은 쌍을 본 심판 3명의 판정은 서로 상관돼 있다. 상관된 것을 독립 취급하면 분산을 과소평가한다 — **CI가 실제보다 좁아진다.** 그리고 이 CI가 retire2의 은퇴 판정을 직접 구동한다. 좁은 CI는 과신이고, 과신은 억울한 조기 은퇴를 만든다.

수정: **쌍(pair) 단위 클러스터 재표집 + 1000회 고정.** 쌍을 뽑으면 그 쌍의 심판 판정 전부가 함께 들어간다.

```javascript
for(let i=0;i<N;i++){
  const p = pairs[rand()*N | 0];               // 쌍 단위 복원추출
  for(const jd in p.juryScore)                  // 그 쌍의 심판 판정을 '함께'
    res.push({a:p.a.model, b:p.b.model, s:p.juryScore[jd]});
}
```

이 수정이 실측 리더보드를 어떻게 바꿨는지는 다음 편에서 — 스포일러: 순위는 안 바뀌고, CI가 평균 51% 넓어지고, "확실해 보였던" 결론 하나가 사라진다.

## 결함 3 — 못 읽은 판정이 조용히 무승부가 되고 있었다

심판 응답에서 승자를 뽑는 파서는 `Winner: A` 형식을 정규식으로 잡는다. 못 잡으면? **조용히 tie(0.5)로 처리된다.** 예컨대 `Winner is A`라고 쓰면 매치가 안 되고, 그 판정은 아무 흔적 없이 무승부로 집계에 들어간다. 파이썬 쪽은 파싱 성공 여부(parse_ok)를 기록하는데 JS 쪽엔 그 텔레메트리가 없었다. temperature 0.7(결함 1)로 형식 이탈이 잦았다면, 보이지 않는 손이 결과를 0.5 쪽으로 당기고 있었을 수 있다.

여기서 유혹은 "정규식을 넓히자"인데, 일부러 안 넓혔다. 과거에 `\b` 없는 느슨한 정규식이 본문의 "the winner: **a** close call"에서 'a'를 A 승리로 오파싱하던 버그가 실제로 있었기 때문이다. 느슨한 파서는 침묵 오염을 다른 침묵 오염으로 바꿀 뿐이다.

대신 **실패를 보이게** 만들었다: 파싱 성공/실패를 집계해 실패 시 로그에 즉시 경고하고, 결과 화면 체크리스트에 파싱 성공률을 상시 노출한다. 고칠 수 없으면 최소한 셀 수는 있어야 한다.

## 결함 4 — 그 빈 catch 블록 (4편의 범인, 이번엔 근본 수정)

4편 크래시의 원인이던 localStorage 침묵 실패. 이번에 용량을 실측해보니 쌍당 약 13KB였다 — 동화 2편(~4KB) + 심판 3명 × 양방향 근거 전문(~9KB). 10MB 한도면 775쌍 근처가 임계점이고, 실제 797쌍 체크포인트 파일이 10.02MB였다. 4편 크래시는 우연이 아니라 산수였다.

수정: 저장 실패 시 **IndexedDB(사실상 무제한)로 즉시 전량 폴백**하고, 앱을 켤 때 IndexedDB 쪽이 더 완전하면 자동 복구해 이어하기 지점을 잡는다. localStorage(빠름) → IndexedDB(무제한) → GitHub(원격) 3중이 됐다.

## 결함 5 — 난이도 절대값은 처음부터 못 믿을 물건이었다

4편 끝에 "Match Rate 2.9%는 도구의 한계"라고 했다. 그 한계의 정체: 텍스트 난이도를 가독성 공식(Flesch-Kincaid, Coleman-Liau, ARI 합성)으로 재고 학년→Lexile로 보간하는데, 이 공식들은 원래 긴 지문용이다. 문장 몇 개짜리 동화에선 희귀 장음절어 하나에 학년이 튀고, 공인 보정(MetaMetrics) 없는 매핑이라 절대값이 계통적으로 높게 나온다. 목표 500L짜리 동화를 재면 중앙값 868L이 나오는 이유다.

전면 재설계 대신 두 가지로 정리했다. ① 이상치가 평균·분산을 왜곡하지 않게 합성 학년을 [0,14]로 클램프(로버스트화). ② **절대값 대신 반응성 β를 주 지표로** — "레벨을 한 단계 낮춰달라고 하면 실제로 한 단계 쉬워지는가"의 회귀 기울기다. 절대 눈금이 틀린 자로도 **변화량**은 잴 수 있다. 실측에서 β는 전 모델 양수(+1.8~+3.2)로, 모델들이 난이도 지시에 실제로 반응한다는 건 확인됐다.

## 고치다가 또 만난 이중 구현의 저주

결함 2를 고치면서 파이썬 쪽 부트스트랩도 쌍 클러스터로 맞추려 했다. 그러자 **retire 스케줄러의 회귀 테스트가 깨졌다.** 파이썬 retire의 결정적 테스트들(어느 배치에서 누가 은퇴하는지까지 고정)이 기존 contest 단위 CI에 보정돼 있었던 것이다. CI 방식을 바꾸면 은퇴 시나리오가 통째로 바뀐다.

파이썬 수정은 원복했다. 대신 앱(JS)만 개선하고, 두 구현의 CI 방식이 의도적으로 다르다는 사실을 앱 문서에 그대로 적었다. 억지로 맞추다 검증된 파이프라인을 깨는 것보다, **갈라진 상태를 정직하게 문서화**하는 쪽을 골랐다. 완전한 해결은 아니고, 파이썬 retire를 새 CI로 재보정하는 건 남은 숙제다.

- 파이썬 테스트 17+6개 전부 그린 (파이썬 무변경 확인)

## 정리

| # | 결함 | 본질 | 수정 |
|---|---|---|---|
| 1 | 판정 temperature 0.7 | 비결정적 심판 + 구현 드리프트 | 판정만 temp 0 분리 |
| 2 | 부트스트랩 60회·contest 단위 | 상관 무시 → CI 과소 → retire 과신 | 쌍 클러스터·1000회 |
| 3 | 파싱 실패 = 조용한 tie | 침묵 오염, 텔레메트리 부재 | 실패 집계·경고·노출 (정규식은 안 넓힘) |
| 4 | localStorage 빈 catch | 침묵 데이터 유실 | IndexedDB 폴백 + 자동 복구 |
| 5 | 절대 Lexile 미보정 | 도구의 계통 오차를 성능으로 오독 | 클램프 + 반응성 β 중심 |

관통하는 패턴은 하나다. **다섯 개 전부 "조용히" 틀리는 버그였다.** 크래시는 오히려 친절하다 — 진짜 위험한 건 멀쩡한 얼굴로 좁은 CI를, 무승부를, 0점을 돌려주는 코드다. 그리고 이건 만든 사람 눈에는 잘 안 보인다. 심판에게 자기평가를 금지시킨 논리는 코드에도 똑같이 적용됐어야 했다.

다음 편: 결함 2를 고치자 벌어진 일 — 신뢰구간이 51% 넓어지면서 리더보드의 "확실했던" 결론이 하나 사라졌고, 그 대가를 능동 표집으로 되갚는 실험.
