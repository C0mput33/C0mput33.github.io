[4편](/posts/46-dollar-frontier-live-eval-13-models/)의 실측 리포트를 정리하다가 모순 하나를 깨달았다. 이 시스템은 심판 LLM이 자기 계열 모델을 평가하지 못하게 설계돼 있고, 자기 것에 후한 편향이 실측에서 t=6.43으로 검출되기도 했다. 그런데 그 시스템의 코드 자체는 줄곧 만든 사람과 만든 에이전트만 들여다보고 있었다.

그래서 7월 15일, 실측 데이터를 쥔 채로 별도의 AI 에이전트에게 엔진 전체 감사를 맡겼다. 주문은 하나였다. 주석과 문서를 믿지 말고 코드만 볼 것, 칭찬 말고 결함을 찾을 것.

돌아온 지적 중 검증을 통과한 게 5개였다. 하나씩 코드로 재확인하고, 고치고, 테스트를 붙였다.[^commit]

## 결함 1: 심판이 temperature 0.7로 돌고 있었다

API 호출 함수에 <span class="term" data-tip="LLM 출력의 무작위성을 조절하는 값. 0이면 같은 입력에 항상 같은 출력이 나오고, 높을수록 다양하지만 들쭉날쭉해진다.">temperature</span>가 하드코딩돼 있었다.

```javascript
// 수정 전: 생성·번역·판정이 전부 이 함수 → 전부 temperature 0.7
body: JSON.stringify({model, messages, temperature: 0.7, ...})
```

동화 생성은 0.7이 맞다. 창작이니까. 문제는 판정까지 같은 함수를 타면서 0.7로 나갔다는 것이다. 같은 두 동화를 두 번 물으면 다른 승자가 나올 수 있는 비결정적 심판이었다. 더 아픈 건 파이썬 파이프라인의 심판은 처음부터 `temperature: 0.0`이었다는 점이다. 2편에서 말한 이중 구현 드리프트가 또 있었다.

수정은 한 줄 개념이다. 호출에 옵션을 열고, 판정 두 콜(정방향·역방향)만 temp 0으로 보냈다.

```javascript
const r1 = await orCall(judge, SYS, user(A,B), {temp: 0});  // 판정 = 결정적
const r2 = await orCall(judge, SYS, user(B,A), {temp: 0});  // 생성·번역은 0.7 유지
```

## 결함 2: 신뢰구간이 실제보다 좁게 나오고 있었다

리더보드의 95% CI를 만드는 <span class="term" data-tip="가진 데이터에서 복원추출로 여러 번 다시 뽑아 같은 계산을 반복해, 추정치가 얼마나 흔들리는지(불확실성) 재는 방법.">부트스트랩</span>에 문제가 둘 겹쳐 있었다.

반복이 60회였다. 데이터가 클수록 반복을 줄이는 코드(`n>1500 ? 60 : ...`)라, 정작 797쌍 실측에서 60회만 재표집했다. 반복 수가 좁히는 건 데이터 노이즈가 아니라 백분위 끝값 자체의 몬테카를로 노이즈라서, 발상이 뒤집혀 있었다. 60개 표본에서 2.5백분위는 사실상 아래에서 1~2번째 값인데, 극단 순서통계량은 표본이 적을수록 심하게 흔들린다. 끝점이 흔들리는 <span class="term" data-tip="'진짜 값이 이 범위 안에 있을 것'이라고 정해진 신뢰 수준(예: 95%)으로 말할 수 있는 구간. 두 후보의 구간이 겹치면 우열을 단정할 수 없다.">신뢰구간</span>이었던 셈이다.

재표집 단위도 틀렸다. 판정(contest) 하나하나를 독립으로 뽑았는데, 같은 쌍을 본 심판 3명의 판정은 서로 상관돼 있다. 상관된 것을 독립 취급하면 분산을 과소평가하고, CI가 실제보다 좁아진다. 이 CI가 retire2의 은퇴 판정을 직접 구동하니, 좁은 CI는 과신이고 과신은 억울한 조기 은퇴로 이어질 수 있다.

수정은 쌍(pair) 단위 클러스터 재표집과 1000회 고정이다. 쌍을 뽑으면 그 쌍의 심판 판정 전부가 함께 들어간다.

```javascript
for(let i=0;i<N;i++){
  const p = pairs[rand()*N | 0];               // 쌍 단위 복원추출
  for(const jd in p.juryScore)                  // 그 쌍의 심판 판정을 '함께'
    res.push({a:p.a.model, b:p.b.model, s:p.juryScore[jd]});
}
```

이 수정이 실측 리더보드를 어떻게 바꿨는지는 6편에 있다. 순위는 안 바뀌었고, CI가 평균 51% 넓어졌고, 결론 하나가 통계적 동률로 바뀌었다.

## 결함 3: 못 읽은 판정이 조용히 무승부가 되고 있었다

심판 응답에서 승자를 뽑는 파서는 `Winner: A` 형식을 정규식으로 잡는다. 못 잡으면 tie(0.5)로 처리된다. 예컨대 `Winner is A`라고 쓰면 매치가 안 되고, 그 판정은 아무 흔적 없이 무승부로 집계에 들어간다. 파이썬 쪽은 파싱 성공 여부(parse_ok)를 기록하는데 JS 쪽엔 그 텔레메트리가 없었다. temperature 0.7(결함 1)로 형식 이탈이 잦았다면 결과가 0.5 쪽으로 끌려가고 있었을 수 있다.

정규식을 넓히고 싶은 유혹이 있었지만 일부러 안 넓혔다. 과거에 `\b` 없는 느슨한 정규식이 본문의 "the winner: **a** close call"에서 'a'를 A 승리로 오파싱하던 버그가 실제로 있었다. 느슨한 파서는 침묵 오염을 다른 침묵 오염으로 바꿀 뿐이다.

대신 실패를 보이게 만들었다. 파싱 성공과 실패를 집계해 실패 시 로그에 즉시 경고하고, 결과 화면 체크리스트에 파싱 성공률을 상시 노출한다. 고칠 수 없으면 최소한 셀 수는 있어야 한다.

## 결함 4: 그 빈 catch 블록

4편 중단 사고의 원인이던 <span class="term" data-tip="브라우저가 사이트별로 제공하는 기본 저장 공간. 간편하지만 용량이 약 10MB로 제한된다.">localStorage</span> 침묵 실패를 이번에 근본 수정했다. 용량을 실측해보니 쌍당 약 13KB였다. 동화 2편이 4KB쯤, 심판 3명 × 양방향 근거 전문이 9KB쯤이다. 10MB 한도면 775쌍 근처가 임계점이고, 실제 797쌍 체크포인트 파일이 10.02MB였다. 4편의 중단은 우연이 아니라 산수였다.

수정: 저장 실패 시 <span class="term" data-tip="브라우저에 내장된 데이터베이스. localStorage와 달리 용량이 사실상 무제한이다.">IndexedDB</span>(사실상 무제한)로 즉시 전량 폴백하고, 앱을 켤 때 IndexedDB 쪽이 더 완전하면 자동 복구해 이어하기 지점을 잡는다. localStorage(빠름), IndexedDB(무제한), GitHub(원격)의 3중이 됐다.

## 결함 5: 난이도 절대값은 원래 못 믿을 물건이었다

4편 끝에 "Match Rate 2.9%는 도구의 한계"라고 적었다. 그 한계의 정체는 이렇다. 텍스트 난이도를 가독성 공식(Flesch-Kincaid, Coleman-Liau, ARI 합성)으로 재고 학년→<span class="term" data-tip="영어 텍스트의 읽기 난이도를 나타내는 지수(대략 0~1800L). 숫자가 클수록 어렵다.">Lexile</span>로 보간하는데, 이 공식들은 원래 긴 지문용이다. 문장 몇 개짜리 동화에선 희귀 장음절어 하나에 학년이 튀고, 공인 보정(MetaMetrics) 없는 매핑이라 절대값이 계통적으로 높게 나온다. 목표 500L짜리 동화를 재면 중앙값 868L이 나오는 이유다.

전면 재설계 대신 두 가지로 정리했다. 이상치가 평균과 분산을 왜곡하지 않게 합성 학년을 [0,14]로 클램프했고, 절대값 대신 반응성 β를 주 지표로 삼았다. β는 "레벨을 한 단계 낮춰달라고 하면 실제로 한 단계 쉬워지는가"의 회귀 기울기다. 절대 눈금이 틀린 자로도 변화량은 잴 수 있다. 실측에서 β는 전 모델 양수(+1.8~+3.2)로, 모델들이 난이도 지시에 실제로 반응한다는 것까지는 확인됐다.

## 고치다가 다시 만난 드리프트

결함 2를 고치면서 파이썬 쪽 부트스트랩도 쌍 클러스터로 맞추려 했다. 그러자 retire 스케줄러의 회귀 테스트가 깨졌다. 파이썬 retire의 결정적 테스트들(어느 배치에서 누가 은퇴하는지까지 고정)이 기존 <span class="term" data-tip="집계의 최소 단위. 한 쌍에 대한 한 심판의 (양방향 평균) 판정 하나를 말한다.">contest</span> 단위 CI에 보정돼 있었던 것이다. CI 방식을 바꾸면 은퇴 시나리오가 통째로 바뀐다.

파이썬 수정은 원복했다. 앱(JS)만 개선하고, 두 구현의 CI 방식이 의도적으로 다르다는 사실을 앱 문서에 그대로 적었다. 억지로 맞추다 검증된 파이프라인을 깨는 것보다 갈라진 상태를 문서화하는 쪽을 골랐다. 완전한 해결은 아니고, 파이썬 retire를 새 CI로 재보정하는 건 남은 숙제다.

## 검증

다섯 수정 전부에 안전망을 붙였고, 재현 가능하게 남겼다.

- 앱 전체 JS 구문 검증. `<script>` 블록을 추출해 Node의 `vm.Script`로 파싱한다. 3,400줄 단일 파일에선 오타 하나가 앱 전체를 백지로 만들기 때문에 편집마다 도는 1차 관문이다.
- 바뀐 순수 함수의 단위 테스트 13종. 파서는 과거 오파싱 사례들("Winner: Although...", "winner: a close call", `**A**` 볼드)을 회귀 케이스로 고정했고, 부트스트랩은 시드 고정 결정성과 쌍 단위 재표집, 빈 입력을 검사한다.
- 파이썬 테스트 확인. `python3 eval/tests/test_eval.py`(17개)와 `test_schedule.py`(6개) 전부 통과. 파이썬을 안 건드렸다는 사실 자체를 테스트로 확인했다.

## 다섯 결함의 공통점

| # | 결함 | 본질 | 수정 |
|---|---|---|---|
| 1 | 판정 temperature 0.7 | 비결정적 심판 + 구현 드리프트 | 판정만 temp 0 분리 |
| 2 | 부트스트랩 60회·contest 단위 | 상관 무시 → CI 과소 → retire 과신 | 쌍 클러스터·1000회 |
| 3 | 파싱 실패 = 조용한 tie | 침묵 오염, 텔레메트리 부재 | 실패 집계·경고·노출 (정규식은 안 넓힘) |
| 4 | localStorage 빈 catch | 침묵 데이터 유실 | IndexedDB 폴백 + 자동 복구 |
| 5 | 절대 Lexile 미보정 | 도구의 계통 오차를 성능으로 오독 | 클램프 + 반응성 β 중심 |

다섯 개 전부 조용히 틀리는 버그였다. 크래시는 차라리 낫다. 멀쩡한 얼굴로 좁은 CI를, 무승부를, 0점을 돌려주는 코드가 위험하고, 이런 건 만든 사람 눈에 잘 안 보인다. 심판에게 자기평가를 금지시킨 논리를 코드 리뷰에도 적용했어야 했는데, 늦게라도 하길 잘했다.

6편은 결함 2를 고친 뒤 신뢰구간을 다시 계산한 기록이다.
