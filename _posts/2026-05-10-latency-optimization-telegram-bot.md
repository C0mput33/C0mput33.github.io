Telegram 봇의 평균 응답 시간은 4.2초, P95는 8.7초였다. 7일 기준선 로그 218건을 확인한 뒤 멀티 모델 <span class="term" data-tip="오픈라우터가 같은 모델을 여러 서빙 공급자 가운데 가격·가용성 기준으로 골라 보내는 것. 공급자가 바뀌면 캐시가 이어지지 않으므로, 라우팅 분산과 캐시 히트율은 서로 상충한다.">라우팅</span>과 <span class="term" data-tip="반복되는 프롬프트 앞부분의 계산 결과를 재사용해 지연이나 입력 비용을 줄이는 기능. 자동·명시형 여부, 최소 길이, 만료 시간, 할인율은 모델과 공급자마다 다르다.">프롬프트 캐싱</span>을 적용했다. 이후 2주 로그에서 평균은 1.8초, P95는 3.1초로 줄었다. 이 글은 측정 조건, 두 변경의 역할, 아직 남은 오분류와 <span class="term" data-tip="요청한 데이터나 프롬프트 상태가 캐시에 없어 원래 계산이나 저장소 조회를 다시 수행하는 경우. 캐시 키·만료·라우팅이 달라져도 발생할 수 있다.">캐시 미스</span> 영향을 다룬다.

## 기준선 측정

개선 전 7일간 메시지 로그를 분석했다. (n=218)

| 지표 | 값 |
|------|-----|
| 평균 응답 시간 | 4.2초 |
| <span class="term" data-tip="측정값을 작은 순서로 놓았을 때 95%가 이 값 이하에 들어오는 백분위. 응답 시간 P95가 3초라면 요청의 약 5%는 3초보다 느렸다는 뜻이다.">P95</span> 응답 시간 | 8.7초 |
| 일 평균 Claude <span class="term" data-tip="소프트웨어가 다른 소프트웨어의 기능이나 데이터에 접근할 때 따르는 호출 계약. 사용할 주소, 입력 형식, 인증, 응답과 오류 규칙을 함께 정의한다.">API</span> 비용 | $1.24 |
| 전체 메시지 중 단순 Q&A | ~63% |

단순 Q&A 비율이 63%였다. "오늘 뭐야?", "어떤 게 뭐야?" 같은 짧은 질문들. 이걸 전부 Claude Sonnet이 처리하고 있었다.

## 개선 1: 멀티 모델 라우팅

작업 유형을 세 가지로 분류하고 각각 다른 모델로 라우팅했다.

```
FAST_QA  → Gemini Flash 2.0   (응답 시간: ~1.2초, 비용: ~$0.0001/건)
TOOL_TASK → Claude Sonnet 4.5  (응답 시간: ~4.5초, 비용: ~$0.02/건)
DEEP_TASK → Claude Sonnet 4.5  (응답 시간: ~15~40초, 비용: ~$0.10/건)
```

키워드 기반 분류기를 만들었다. 별도 ML 분류기를 두지 않은 이유는 다음과 같다.

1. 분류 자체에 레이턴시가 추가되면 의미가 없다
2. 도메인이 제한적 (캘린더, GitHub, Notion, GCS 키워드가 명확함)
3. 과분류(TOOL_TASK로 잘못 분류)가 과소분류(FAST_QA로 잘못 분류)보다 낫다

실제 분류 결과 (개선 후 2주, n=312):

| 분류 | 비율 | 오분류율 |
|------|------|---------|
| FAST_QA | 65% | ~3% (도구가 필요했는데 빠진 경우) |
| TOOL_TASK | 32% | ~1% |
| DEEP_TASK | 3% | ~0% |

## 개선 2: 프롬프트 캐싱

Claude API의 프롬프트 캐싱 기능을 활성화했다.

<span class="term" data-tip="대화 전체에 적용할 역할·행동 규칙·출력 제약을 모델에 전달하는 상위 지시. 공급자 API에 따라 system 또는 developer 메시지로 표현된다.">시스템 프롬프트</span>(~2,400 tokens)와 tools 정의(~1,800 tokens)를 캐싱 대상으로 지정했다. `cache_control: { type: "ephemeral" }` — <span class="term" data-tip="Time To Live. 데이터에 걸어 두는 유효 시간으로, 지나면 자동 삭제된다. 진행률처럼 잠깐만 의미 있는 값을 별도 청소 코드 없이 관리할 수 있다.">TTL</span> 5분.

캐시 히트 시 토큰당 비용이 **읽기 기준 90% 감소**, 레이턴시도 **~50% 감소**한다.[^1]

실제 <span class="term" data-tip="재사용 가능한 입력 중 실제 캐시에서 읽힌 비율. 계산법과 할인율은 공급자마다 다르므로 cached token 수와 실제 청구액을 함께 확인해야 한다.">캐시 히트율</span>은 대화가 집중적으로 일어나는 시간대 기준 약 **72%**였다. 대화 없이 5분 이상 지나면 캐시가 만료되므로, 사용 패턴에 따라 효과가 달라진다.

## 결과

| 지표 | 개선 전 | 개선 후 | 변화 |
|------|---------|---------|------|
| 평균 응답 시간 | 4.2초 | **1.8초** | -57% |
| P95 응답 시간 | 8.7초 | **3.1초** | -64% |
| 일 평균 API 비용 | $1.24 | **$0.35** | -72% |

측정 기간: 각 2주. 동일한 사용 패턴에서 비교.

## 한계와 트레이드오프

이 수치는 사용 패턴이 달라지면 그대로 유지되지 않는다.

**FAST_QA 오분류**: "이슈 뭐야?"는 TOOL_TASK가 맞는데 FAST_QA로 분류되면, Gemini가 "어떤 이슈를 말씀하시나요?" 같은 애매한 답을 준다. 이 경우 재질문이 발생해서 오히려 총 시간이 늘어난다.

**캐시 미스**: 5분 넘게 대화가 없다가 새 메시지가 오면 캐시 미스. 첫 번째 메시지는 항상 캐싱 비용이 붙는다. 자주 쓰지 않는 날은 비용 절감 효과가 적다.

**Gemini vs Claude 응답 품질 차이**: FAST_QA라도 미묘한 맥락이 필요한 질문에서 Gemini가 더 피상적으로 답할 때가 있다. 이건 아직 해결하지 못했다.

## 다음 최적화 방향

현재 P95가 3.1초인데, TOOL_TASK가 여기서 지배적이다. GitHub API나 Google Calendar API 호출 자체가 0.5~1.5초를 차지한다. 이건 모델 선택으로는 개선이 안 된다.

다음 단계로 고려 중인 것: **병렬 도구 호출**. 이미 `asyncio.gather`로 일부 구현되어 있는데, 독립적인 도구들을 병렬로 실행하면 P95를 더 줄일 수 있을 것 같다.
